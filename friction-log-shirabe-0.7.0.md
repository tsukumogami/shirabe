# Friction log — shirabe 0.7.0

Observations during the /scope -> /brief -> /prd -> /design -> /plan chain run for `shirabe-child-dispatch-contract`. One paragraph per observation; lead with friction, then root cause, then fix suggestion; tag with skill; BUG/SMELL flags.

## /design friction

The /design skill's reference flow (`phase-0-setup-prd.md` and the Phase 1-6 phases) assumes the design-running agent has access to spawning sub-agents for the decision-rosters / decision-researchers loop (Phase 2) and for the 3-or-5 reviewer jury (Phase 4-equivalent). In this run I was invoked as a single dispatched child of /scope under v1's `team_primitive: single-team-per-leader-no-nested`, which forbids nesting a sub-team inside the dispatched agent. The friction: I walked all six decisions (D1 mechanism, D2 declarator, D3 construction layer, D4-D6 D5 backlog items) and all four jury reviewers inline within my own agent context. Root cause: the dispatch contract being designed by this very run is the unresolved contract — the substrate today does not give a dispatched child a clean way to spawn sub-agents, so the only honest implementation is inline decision-protocol walks. Fix suggestion: the new DESIGN explicitly codifies this (children construct their own teams; the parent does not, and a dispatched-child running /design walks the decision protocol inline because nesting is forbidden). This is meta-evidence — the very work designing the dispatch contract had to dispatch under the unresolved contract — and the DESIGN's Decision 1's option 1C is the recipe by which this run actually executed. SMELL.

The /design skill's Phase 4-equivalent in this dispatched-child context could not produce per-reviewer verdict files at conventional `wip/research/design_<topic>_phase4_<reviewer>.md` paths because spawning four sub-agents (one per reviewer) violates `single-team-per-leader-no-nested`. The friction: the jury PASS/PASS/PASS/PASS verdict lives only in the dispatched agent's transcript, not on disk. Root cause: the substrate constraint + the absence of a "single-agent runs N inline reviewer passes and writes N verdict files" pattern in /design's references. Fix suggestion: extend /design's Phase 4 reference to recognize the dispatched-child execution mode (when invoked from /scope or /charter), and document an "inline reviewer pass, single verdict-summary file" fallback that satisfies AC18-style legibility audits without requiring sub-agent spawning. The fallback is already what dispatched-child runs implicitly do; it should be named. SMELL.

The /design skill's Phase 0 PRD-mode setup script reads `$ARGUMENTS` to locate the PRD path, but this run was invoked via `Skill design shirabe-child-dispatch-contract` (the topic slug, not a path). The setup phase had to derive the PRD location from the slug. Friction: minor — the convention `docs/prds/PRD-<topic>.md` resolved cleanly — but the phase reference explicitly says "Read the PRD file from the path provided in `$ARGUMENTS`," which is misleading when /design is dispatched by /scope with a slug. Root cause: /design's Phase 0 docs were written for direct invocation, not for dispatched-child invocation. Fix suggestion: add a "Topic-slug input mode" branch to `phase-0-setup-prd.md` mirroring the way /brief and /prd handle slug-only invocation under the parent_orchestration sentinel. SMELL.

## Meta-observation on the dispatch contract under design

The harness used to dispatch this /design run from /scope is the implicit answer to issue #150 for this run. /scope invoked me via the Skill tool (inline) with the topic slug as $ARGUMENTS; I read the parent_orchestration sentinel from the state file; I ran the entire decision-protocol walk and the four jury reviewers inline within my own agent context; I did not spawn any sub-agents (no Agent tool, no TeamCreate, no Task tool); I wrote the DESIGN artifact and committed it; /scope will read my durable artifact at `docs/designs/DESIGN-shirabe-child-dispatch-contract.md` when control returns. This IS Decision 1's option 1C in practice. The DESIGN codifies what this run actually did. The friction is that the codification is needed in the first place — orchestrator agents reading the v1 docs cold cannot derive this behavior; they reach for one of the rejected options and the author flags the mismatch. SMELL (not BUG, because the v1 implementation works; the legibility is what's broken).
