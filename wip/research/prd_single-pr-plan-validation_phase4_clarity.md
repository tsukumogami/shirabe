# Clarity Review

## Verdict: PASS

The PRD is unusually specific for its altitude -- requirements name file paths, struct names, exact check codes, and verbatim notice content shapes; the few residual ambiguities are minor wording choices rather than gaps that would let two developers build different things.

## Ambiguities Found

1. R2 "malformed dependencies declaration (per the format reference at `skills/plan/references/quality/plan-doc-structure.md`)": the PRD defers the malformed-shape definition to an external file -> two developers may read that reference and disagree on what counts as "malformed" if the reference is ambiguous itself -> either enumerate the malformed shapes inline (missing colon, non-comma-separated list, wrong indentation, etc.) or pin a specific section/anchor in the reference (`#dependencies-grammar`) so the contract is unambiguous.
2. R3 "outline key, as defined by the format reference": the outline-key definition is delegated -> a developer must read another file to know whether the key is the heading text verbatim, a slug derived from the heading, or the trailing identifier (`Outline 1`, `outline-1`, `outline 1`) -> state the key extraction rule inline (e.g., "the trailing identifier after `### Outline ` lowercased and hyphenated") or quote the reference's canonical definition.
3. R5 "non-comment, non-empty entity rows after the header/separator": "non-comment" is undefined in markdown-table context -> the validator implementer may guess whether HTML comments inside cells, lines starting with `<!--`, or rows whose first cell is `<!-- placeholder -->` count as comments -> define the comment-row shape explicitly (e.g., "a row whose first non-whitespace cell content begins with `<!--`").
4. R9 "include enough information to make the fix mechanical": "mechanical" is subjective -> a reviewer cannot binarily verify whether a notice's wording is mechanical-enough -> replace the adjective with the existing enumerated requirements (the bullets already cover this) and drop the "mechanical" framing, or add an example notice string per bullet to anchor the bar.
5. R11 "The exact field shape (a map, a vec of tuples, an optional sibling struct) is a downstream implementation choice": this is deliberate deferral and is fine for a PRD, but the phrase "extends the FormatSpec contract" could be read as either "add fields to FormatSpec" or "introduce a sibling type that FormatSpec composes" -> clarify the contract-level requirement (FormatSpec gains the capability to express execution-mode-aware required sections; the in-memory representation is downstream).
6. Sub-check B AC "An outline block with a malformed `Dependencies:` declaration produces an FC14 notice naming the outline key and the malformed shape": "malformed shape" is not enumerated in the AC -> a test author cannot derive a finite test matrix from "malformed" -> either reference R2's enumerated malformed cases or list 2-3 canonical malformed shapes (missing colon, bare token without commas, embedded newline).
7. Sub-check D AC "the count of outline blocks in `## Issue Outlines`": the count rule for outline-block-with-empty-body or outline-block-that-is-only-a-heading is unspecified -> developers could count or not count heading-only stubs -> state whether a heading-only outline counts toward `issue_count` (recommend "yes -- a heading-only outline counts as one outline block, and its missing-goal/missing-AC defects fire as separate FC14 notices").
8. Implementation-level AC "An existing single-pr plan in the corpus that previously passed validation may now produce FC14 notices; the validator exits 0": "may now" is conditional but unverifiable as a binary pass/fail -> reframe as a verifiable claim ("at least one plan in `docs/plans/` produces an FC14 notice after this PR" if true, or simply drop this AC since R7 + R15 already cover the exit-0 contract).
9. R12 / Implementation-level AC "total over arbitrary input": "arbitrary input" is unbounded -> a reviewer cannot binarily verify totality across all possible inputs -> reframe as "total over the corpus plus a fuzz test of N random-byte inputs that do not panic" or "total over the test-corpus shapes enumerated in R13 plus an explicit list of pathological cases (empty file, single line, unterminated code fence, mismatched headings)."

## Suggested Improvements

1. Inline the outline-block grammar (Goal, Acceptance Criteria, Dependencies field shapes with exact heading/marker text): the PRD currently routes through `plan-doc-structure.md` for the grammar; pulling the 5-10 line grammar inline would make this PRD self-contained for the implementer and reviewer without re-opening the reference.
2. Add one example notice message per FC14 sub-check (the Decision D2 "distinguishing prefix text" already hints at shape, but examples like `[FC14] outline outline-3 missing goal` would let the implementer and reviewer match strings verbatim in tests): R9 enumerates required information; example strings would make Sub-check ACs binarily verifiable.
3. State the dependency-token grammar explicitly (comma-separated identifiers; whitespace; case-sensitivity): R3 names "dependency token" but does not say whether `outline-1, outline-2` and `outline-1,outline-2` and `Outline-1` are all valid; the implementer must pick.
4. Tighten the "issue_count mismatch" AC to distinguish "missing frontmatter field" from "field present but wrong": both produce different developer experiences and the PRD doesn't say if missing `issue_count` itself fires FC14 or some other code.

## Summary

The PRD is concrete and implementer-ready: paths, structs, check codes, severity wiring, and per-sub-check acceptance criteria are all named explicitly, and most ACs are objectively verifiable. The residual ambiguities are about grammar precision (what counts as "malformed" or "outline key") rather than scope or intent -- two developers would build the same FC14 check, though their notice strings and edge-case handling for malformed inputs might differ. Tightening the inline grammar references and replacing "mechanical"/"arbitrary input" with enumerated criteria would convert the PASS into a stronger one.
