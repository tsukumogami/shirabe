# Security Review: work-on-koto-unification

## Dimension Analysis

### External Artifact Handling

**Applies:** No

This design does not download, execute, or process external inputs from untrusted sources. All inputs are authored by the same user operating the workflow:

- PLAN documents are local markdown files written by the user or generated by prior workflow phases within the same session.
- The manifest JSON is created and consumed entirely within the local workflow. It is never fetched from a remote source.
- Koto context keys store locally-produced summaries and review results, not external content.
- The queue script (`plan-queue.sh`) reads local JSON and produces local JSON. It does not fetch URLs, download artifacts, or invoke remote APIs.

The closest thing to "external" input is GitHub issue metadata (titles, numbers) referenced from PLAN docs, but this data is used as labels and routing keys, not executed or interpreted as code.

### Permission Scope

**Applies:** Yes (low severity)

**Risks identified:**

1. **Filesystem writes via manifest and koto state.** The orchestrator writes manifest files to `wip/` and koto writes state files to its configured state directory. Both are scoped to the project working directory. No writes occur outside the repository tree.

2. **Shell script execution.** `plan-queue.sh` runs as a shell script with the invoking user's permissions. It uses `jq` for JSON processing. The script does not use `eval`, does not interpolate untrusted strings into shell commands, and does not require elevated privileges. The design explicitly states it "doesn't execute arbitrary commands."

3. **Multi-agent spawning.** Review panels spawn 3 parallel agents for scrutiny and code review. These agents operate within Claude Code's existing permission model -- they don't gain additional filesystem, network, or process permissions beyond what the parent session has. No privilege escalation occurs.

**Severity:** Low. All operations stay within the existing permission boundary of a Claude Code session operating on a local repository.

**Mitigations (already present):**
- Script reads/writes only the manifest file path it receives as an argument.
- Koto state files follow established conventions with no new filesystem scope.
- No `sudo`, no network listeners, no daemon processes.

**No additional mitigations needed.**

### Supply Chain or Dependency Trust

**Applies:** Yes (low severity)

**Risks identified:**

1. **jq dependency.** The queue script adds a runtime dependency on `jq` for plan-backed execution. The design acknowledges this in its "Negative Consequences" section. `jq` is a well-established, widely-used tool, but it is an external binary that must be present on the user's system. If a user has a compromised `jq` on their PATH, the queue script would invoke it.

2. **Koto v0.6.0 dependency.** The design requires koto v0.6.0 for structured gate output. Koto is a first-party tool (same organization), so the trust boundary is internal. The strict-mode migration removes the `--allow-legacy-gates` flag dependency, which is a positive supply chain decision -- it avoids relying on a deprecated compatibility shim.

**Severity:** Low. The `jq` dependency is standard tooling, not a novel or exotic dependency. Koto is first-party.

**Mitigations (already present):**
- The design notes "jq is widely available and already used in the workspace for other scripts," meaning this isn't a new trust boundary.
- No new package registries, no new download sources, no new binary fetches introduced.

**No additional mitigations needed.**

### Data Exposure

**Applies:** No

This design does not access or transmit user credentials, secrets, API keys, or personal data:

- Koto context stores workflow artifacts: implementation summaries, review panel results, plan metadata. These are developer workflow data, not sensitive user information.
- The manifest tracks issue numbers, titles, statuses, and dependency relationships -- all of which are already public GitHub data for public repositories.
- No telemetry, no network transmission of workflow state, no logging to external services.
- The `wip/` artifacts are cleaned before PR merge per existing convention, so intermediate state doesn't persist in the main branch.
- The design's Security Considerations section explicitly notes "No credentials or secrets are stored in context."

## Recommended Outcome

**OPTION 3: N/A with justification**

This design is a workflow orchestration refactoring that reorganizes how existing components (koto state machines, skill markdown, review panels) interact. It introduces no new trust boundaries, no new external inputs, no new permission requirements, and no new data flows. The two low-severity items identified (filesystem writes within the repo, jq as an existing dependency) are already mitigated by the design and don't warrant design changes or additional documentation beyond what the design already includes in its Security Considerations and Negative Consequences sections.

## Summary

This design presents minimal security risk. It restructures internal workflow orchestration without introducing new attack surfaces, external dependencies, or data exposure paths. The filesystem and dependency observations are low-severity and already addressed in the design document's own security and consequences sections. No design changes or additional security documentation are needed.
