# Lead: Where should "capstone session state" live, and who computes/drives the merge order?

## Findings

### 1. What /plan's dependency graph already produces (and whether it can drive merge order)

/plan already emits machine-readable ordering data, not just a human Mermaid diagram. Three layers exist:

- **Mermaid Dependency Graph + Implementation Sequence sections** in the PLAN doc. Per `skills/plan/references/quality/plan-doc-structure.md` (required sections 6 and 7), every PLAN carries a "Dependency Graph -- Mermaid diagram showing issue relationships" and an "Implementation Sequence -- critical path and parallelization opportunities." Phase 5 (`skills/plan/references/phases/phase-5-dependencies.md`) computes this: step 5.2 "Build Dependency Graph", 5.3 "Identify Parallelization", 5.4 "Calculate Critical Path", 5.5 "Validate Sequencing" (explicitly checks "No circular dependencies", "All blockers are in the issue list"). The artifact emits an "Issue Dependencies" table with `Blocked By` / `Blocks` columns plus a "Critical Path" line. So /plan already produces a validated DAG with a topological backbone.

- **`plan-to-tasks.sh` emits a true machine-consumable DAG.** Per `skills/plan/references/plan-to-tasks-contract.md`, the script reads a PLAN.md and emits a JSON array where each task has `{"name", "vars", "waits_on": [...]}`. `waits_on` is exactly the edge set a topological sort needs. Quote (multi-pr): `waits_on = list of issue-<M> for each #M in the Dependencies cell; [] if cell is None`. It also exits 2 on "unresolvable dependency reference," so the emitted graph is guaranteed closed and acyclic-checkable. This is the existing bridge from PLAN -> koto task entries.

- **The `**Files**:` star-topology hint.** `plan-doc-structure.md` lines 176-189: `plan-to-tasks.sh parses this field and wires a star topology to prevent concurrent overwrites: the first outline to declare a file becomes its owner; every later outline that shares that file gets a waits_on edge pointing to the owner.` This is a write-conflict ordering mechanism — directly analogous to what a cross-repo merge order needs, but currently scoped to files within one repo, not PRs across repos.

**Verdict on the lead's sub-question:** /plan produces a topological ordering of *issues within a single plan/repo*, and `plan-to-tasks.sh` already serializes it as `waits_on` edges. It does **not** currently model *repos* or *PRs* as nodes, and it does not emit a cross-repo PR merge order. The data shape (DAG of `waits_on` edges) is the right primitive to reuse, but a capstone needs a higher-altitude graph whose nodes are per-repo PRs, derived from which repo each issue lands in.

### 2. The single-pr orchestrator is the existing single-repo precursor to the capstone

`skills/work-on/SKILL.md` "Plan Mode": when `$ARGUMENTS` is a PLAN.md, work-on "runs as a plan orchestrator rather than working on a single issue. Plan mode coordinates multiple per-issue child workflows and assembles a combined PR after all children complete." It derives `impl/<slug>` from the PLAN filename, reuses an existing branch+PR if present (`status: override`), and spawns plan-backed child workflows. This is precisely the capstone pattern (one PR updated across many child steps, consumed before merge) — but bounded to **one branch / one repo**. The capstone generalizes "one shared branch + combined PR" to "one PR per repo + an explicit merge order across them." The orchestrator already owns child sequencing via koto `waits_on`; it is the natural owner to extend.

### 3. Candidate homes for "active capstone" state

**(a) A `wip/` artifact** — REJECTED as the durable home. The workspace CLAUDE.md and `public/shirabe/CLAUDE.md` ("Intermediate Storage") both define `wip/` as non-durable: files "MUST NOT be referenced from any committed final artifact" and "MUST be removed from the branch before a PR can merge." The wip-hygiene rule makes wip/ structurally wrong for "a capstone is active" state, because (i) the capstone outlives any single child session and (ii) the capstone PR/branch is itself a committed durable artifact that would need to *reference* the active-capstone marker — a committed reference to a wip/ path is exactly the dangling-pointer violation the rule names. wip/ remains correct for *transient per-step scratch* (e.g. `wip/plan_<topic>_*.md`), just not for the session-spanning "capstone active" flag. Note shirabe's koto-driven workflows already replace wip/ with koto context (`koto context add`) for reviewable/traceable artifacts — that is the more likely durable store for capstone session context than wip/.

**(b) niwa workspace state (`instance.json` / `.niwa/`)** — REJECTED, and notably so. niwa's `InstanceState` (`internal/workspace/state.go`, schema v4) tracks workspace identity, managed files, repos, overlay, and auth-source audit — it is workspace *configuration* state, not workflow state, and has no field for in-flight work. More decisively, **the niwa mesh/coordination surface is being deleted**: `niwa/docs/designs/current/DESIGN-niwa-mesh-removal.md` (status: Current) removes `internal/mcp`, the coordinator session registry, `niwa session register`, the change-review surface, and the per-worktree daemon. After removal "niwa retains no MCP, no agent-coordination channels, and no change-review web UI." The only retained per-worktree state is `<worktree>/.niwa/attach.state` (`internal/worktree/attach_state.go`) — a PID-based attach lock, written on attach and removed on clean exit, explicitly NOT durable session-workflow state. So the mesh-delegation tools listed in the workspace CLAUDE.md (`niwa_delegate`, `niwa_ask`, etc.) cannot be assumed to exist as a state home going forward. niwa gives us the worktree lifecycle primitive and instance.json, nothing more.

**(c) The capstone branch / PR itself (the PLAN/overarching-plan doc on the branch)** — STRONGEST candidate for the durable anchor. This is durable (committed to a branch), survives any context reset (re-discoverable via `git`/`gh`), is cross-repo-visible if the capstone PR lives in one coordinating repo and lists the child PRs, and complies with wip-hygiene (it is a durable artifact, not a wip/ file). It mirrors the PLAN doc's existing role: it already holds the dependency graph + implementation sequence, and the single-pr orchestrator already re-discovers its branch+PR by matching `impl/<slug>` and an open PR. A capstone-flavored PLAN-on-branch would carry: the overarching plan, the per-repo PR list, and the merge order. Tension: PLANs are *ephemeral and consumed/deleted before merge* (see section 4).

**(d) A CLAUDE.md header** — REJECTED for *active-session* state, viable only for *static convention*. CLAUDE.md headers (e.g. `## Execution Mode:`, `## Repo Visibility:`) are the established home for immutable/default config that skills read at invocation time, and a capstone could define a `## Capstone:` convention header for the *operating contract* the author keeps re-pasting. But headers are durable repo config, not per-session mutable state; writing "capstone X is active right now" into CLAUDE.md would churn a committed config file per session and conflate convention with state. Use CLAUDE.md for the contract/convention; use the branch/PR for the live pointer.

### 4. Who should own merge-order computation, and the ephemerality tension

**Owner: the /plan step computes the order; a /work-on-style capstone coordinator drives (executes) it; niwa provides the worktree primitive only.**

- /plan already owns dependency/sequence computation and already serializes it (`plan-to-tasks.sh` `waits_on`). Extending /plan (or a capstone-aware planning pass) to (i) tag each issue/outline with its target repo and (ii) collapse the issue DAG into a per-repo PR DAG is the lowest-new-machinery path. Inputs it needs: the per-issue repo assignment (today implicit), the existing `waits_on` edges, and the `≤1 worktree/PR per repo` constraint to fold all of a repo's issues into one PR node.
- The capstone coordinator (a cross-repo generalization of work-on Plan Mode) drives execution: spawn ≤1 worktree per repo (via `niwa worktree`), run the per-repo child work, then merge child PRs in the computed topological order, capstone PR last.
- niwa should NOT compute merge order — post-mesh-removal it has no workflow-graph knowledge and is deliberately being reduced to worktree + config. Putting orchestration logic in niwa fights the mesh-removal decision.

**Reconciling "PLAN is consumed before merge" with "state must persist during the session":** these are not in conflict once you separate two roles. (1) The *durable session anchor* is the capstone branch+PR (candidate c) — it persists for the whole multi-step session and is re-discoverable after any context reset by `gh pr list`/branch-match, exactly as work-on Plan Mode already re-discovers `impl/<slug>`. (2) The *plan document* is the ephemeral, consumed-before-merge artifact (per `plan-doc-structure.md`: "PLANs are ephemeral... deleted from the tree in the same commit set that transitions the upstream BRIEF, PRD, and DESIGN to their terminal states"). The capstone's "fully consumed before merge" property is the *same* verify-then-delete forcing function the existing PLAN lifecycle uses — the cascade deletes the plan content but the *PR/branch* (the anchor) is what merges last. So: persist the live pointer as the PR/branch (durable until merge), persist transient step scratch in koto context or wip/, and let the consumed-before-merge plan body be deleted by the cascade. The "capstone active" flag is therefore "an open capstone PR exists on a known branch," which is naturally durable and naturally cleared by the merge — no separate mutable state store is needed.

## Implications

- Reuse, don't invent: `plan-to-tasks.sh`'s `waits_on` JSON and the `**Files**:` star-topology are the right primitive; the new work is lifting the graph one altitude (issue-node DAG -> per-repo-PR DAG) and adding repo assignment to issues/outlines.
- Do not anchor capstone state in niwa: the mesh-removal design (Current) deletes the coordination surface the workspace CLAUDE.md advertises. Design the capstone to depend only on `niwa worktree` + `git`/`gh`, never on `niwa_delegate`/mesh tools.
- The capstone coordinator is a cross-repo generalization of `work-on` Plan Mode, which already coordinates child workflows and re-discovers its branch+PR — extend that, don't build parallel orchestration.
- "Capstone is active" should be a derived fact (open capstone PR on a known branch), not a stored flag — this makes context-reset recovery a `gh pr list` query and makes cleanup automatic on merge.

## Surprises

- The niwa mesh (and the entire `niwa_delegate`/`niwa_ask`/coordinator-registry surface named in the workspace CLAUDE.md) is being removed wholesale (`DESIGN-niwa-mesh-removal.md`, status Current). Any capstone design that leans on niwa mesh state is building on a deleted foundation. The workspace CLAUDE.md's "Channels" section is describing a surface on its way out.
- work-on Plan Mode is already most of a single-repo capstone: one derived branch (`impl/<slug>`), one combined PR, child workflows sequenced by `waits_on`, branch/PR re-discovery via `status: override`. The capstone is largely "the same, but the node is a repo-PR and there are N of them."
- The exact write-ordering problem a cross-repo merge order solves already has a single-repo analogue shipped: the `**Files**:` star topology (first declarer owns, others `waits_on` the owner). Same shape, different altitude.

## Open Questions

- Where is per-issue *repo assignment* recorded today? Single-pr outlines and multi-pr tables have no repo column; a cross-repo capstone needs each issue/outline tagged with its target repo to fold issues into per-repo PR nodes. (Likely a new field on the outline/table.)
- Which repo hosts the capstone PR (the "merged LAST" coordinating PR)? A dedicated coordination repo, or the workspace overlay, or one of the participating repos? This determines cross-repo visibility of the anchor.
- Does koto context (shirabe's koto-driven replacement for wip/) span repos, or is it per-workflow/per-repo? If per-repo, the capstone needs a cross-repo context store for the few session-spanning facts not encoded in the PR/branch.
- How is the per-repo-PR DAG validated for cross-repo cycles (repo A's PR waits on repo B's, and vice versa)? Phase 5's acyclicity check is intra-plan; cross-repo needs an equivalent guard.

## Summary

/plan already computes a validated, topologically-orderable issue DAG and serializes it as `waits_on` edges via `plan-to-tasks.sh`, so the merge-order primitive exists — the only gap is lifting it from an issue-node graph to a per-repo-PR graph by tagging issues with their target repo. The durable home for "a capstone is active" should be the capstone branch/PR itself (re-discoverable after any context reset, wip-hygiene-clean, and naturally cleared on merge), NOT a wip/ file, NOT niwa state — and notably the niwa mesh/coordination surface the workspace advertises is being deleted (`DESIGN-niwa-mesh-removal`, Current), so the capstone must depend only on `niwa worktree` + git/gh. Merge order should be computed by the /plan step (which already owns sequencing) and driven by a cross-repo generalization of work-on's existing Plan-Mode orchestrator; the "consumed before merge" tension dissolves because the persistent anchor is the PR/branch while the ephemeral, cascade-deleted thing is the plan body.
