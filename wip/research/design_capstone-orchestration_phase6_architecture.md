# Architecture Review

**Verdict:** PASS

The decisions cohere around a single pull-based, repo-local axiom; all cited seams exist and behave as described; rejected alternatives carry genuine trade-offs; and the load-bearing PRD requirements are satisfied with the only minor gaps being under-specification a `/plan` decomposition can still proceed from.

## Issues Found

1. **Decision C mislocates `check_issue_closed` (minor, wording).** The text says "the `finalize.rs` `check_issue_closed`'s single-`origin` assumption is relaxed," but `check_issue_closed` lives in `skills/work-on/scripts/run-cascade.sh` (lines 120-154), where it validates the parsed owner/repo against `git remote get-url origin` and rejects any mismatch. `finalize.rs` is the in-progress Rust port of that cascade (its module header notes "the `run-cascade.sh` refactor are later issues"). The seam and the single-origin assumption are both real; only the file attribution is loose. Fix: name run-cascade.sh as the current home and finalize.rs as the port target, so /plan doesn't go looking for the function in the wrong file.

2. **Non-PR serialization gates (R12) are under-specified for the "two-node DAG" model (medium).** The design repeatedly calls the merge order a "two-node DAG (PR nodes + non-PR gate nodes)" and says gate nodes represent things like a package publish between repos, but it does not say how a gate node's satisfaction is determined at `gate`-recompute time. F4 makes PR nodes verifiable via live `gh api` merge state; a publish-step gate has no equivalent authoritative `gh` signal. The PRD's own Known Limitations flag hard-vs-advisory gate enforcement as a DESIGN decision, and this DESIGN leaves it open. Fix: have `references/capstone-strategy.md` state whether a non-PR gate is advisory (annotation only, never blocks the CI gate) or hard (and if hard, what signal `gate` reads). Advisory is the buildable default and keeps F4's fail-closed property intact.

3. **R16 atomicity detection lacks a stated mechanism (medium).** The design asserts a cross-repo atomic-merge requirement is "detected at planning time (a dependency that would require two repos to merge simultaneously) and refused," but the contraction step only produces a `(repo, pr_group)` DAG; a mutual cross-repo dependency surfaces there as a *cycle*, which Decision D/E already route to split-re-sequence-stack resolution. The design does not distinguish "resolvable contraction cycle" from "irreducible atomic-merge requirement that must be refused." Without that test, R16's refusal path and R13's auto-resolution path collide. Fix: define the discriminator (e.g., a cross-repo edge whose two endpoints are in different repos and cannot be made compatible-intermediate => refuse; same-repo or sequenceable => resolve) in the collapse-step spec so /plan can implement both branches.

4. **R22 re-derivation with already-merged nodes is asserted but not specified enough to build (low-medium).** Consequences and F7 both say re-derivation "treats merged PRs as fixed nodes and only re-orders the unmerged remainder" and that this cannot reintroduce a cycle. That is the correct invariant, but the design does not state where the "merged" fact is sourced at re-derivation time (presumably the same live `gh` query as F4) nor what happens if a newly-edited PLAN introduces a dependency *into* an already-merged node (a backward edge that is now unsatisfiable). Fix: state that re-derivation reads merge state live, and that an edge from an unmerged node into an already-merged fixed node is vacuously satisfied while an edge requiring an already-merged node to wait is a halt-with-diagnostic (R21). Buildable without this, but /plan will have to invent the rule.

## Strawman Check

The rejected alternatives are genuine, not strawmen — each names a real capability the chosen option gives up and a concrete cost:

- **A (embed contract in each consumer / standalone tool):** Genuine. "Embed in each consumer" is the real default most teams would reach for, and the drift cost is evidenced by the verified `parent-skill-pattern.md` actually being shared by `/scope` and `/charter`. "Standalone tool" is the alternative the brief explicitly weighed; rejecting it as re-implementing `/scope`+`/work-on` machinery is a substantive architectural call.
- **B (inline `gh` in skill prose / webhooks):** Strong. The inline-prose option is a real, cheaper path; rejecting it as "untestable index-rewrite + order-recompute as markdown" is exactly the kind of logic shirabe has demonstrably moved into the Rust validator. The webhook option is the genuine push-model competitor and the rejection cites four concrete costs (standing infra, cross-repo write-auth, visibility risk, can't attach to forward-declared PRs) plus the R19 violation — not a knocked-down version.
- **C (full cross-repo write walk):** Genuine and the hardest trade. A write walk is the *only* option that could deliver true atomicity, and the design says so plainly before rejecting it on R21 partial-state and push-access grounds. Steelmanned, then refused.
- **D (canon only in PR body / only in PLAN):** Both real and mutually exclusive, and the rejection of each is load-bearing on a verified fact — R8 deletes the PLAN before merge (so PLAN-only loses the data when the gate needs it) and body-only loses authoring-time validatability. This is the sharpest pair in the document.
- **E (separate post-`/plan` grouping layer):** Genuine but lighter. The rejection (duplicates DAG machinery `/plan` owns, splits sequencing ownership) is a reasonable cohesion argument; this is the least adversarial of the set but still a real fork.
- **F (all flags / all preferences):** Genuine; each horn maps to a stated goal it violates ("express once" vs. "config before first use"), and the chosen flag>header>default stack is a verified existing precedent.

## Summary

The design holds together: a single "pull-based and repo-local" axiom drives all six decisions without contradiction, and the load-bearing PRD requirements are satisfied — R19 (no service) and R21 (no partial state) follow directly from the no-cross-repo-write stance; R7/R14 merge-last is enforced by the verified `lifecycle.yml` strict-mode `draft==false` gate recomputing live (F4); R8 consume-before-merge is preserved repo-locally; R13 acyclicity sits at the `/plan` contraction step; R15 visibility is handled by F1 fail-closed redaction. Every cited seam (`gh.rs` Command/.args + charset regex + 4 MiB cap, `finalize.rs` Stop-wall + `is_cross_repo_ref`, run-cascade.sh `check_issue_closed`, `plan-to-tasks.sh` waits_on/`jq --arg`, `lifecycle.yml` strict mode) was confirmed to exist and match its description, and the walking-skeleton sequence is sane (contract ref first, then collapse, then create/sync, then gate, then bindings). The open items — non-PR gate enforcement semantics (R12), the R16-vs-R13 cycle discriminator, and R22 merged-node edge rules — are under-specification rather than incoherence, and a /plan pass can decompose with a one-line resolution of each.
