# Completeness Review

## Verdict: PASS

The PRD is implementable: every Goal and User Story maps to a requirement, every requirement (R1-R13) has at least one corresponding acceptance criterion or is a constraint the ACs collectively enforce, and Out of Scope cleanly fences off the DESIGN-altitude and logic-change creep. The issues below are refinements, not blockers.

## Issues Found

1. **R7 references L01-L07 finding codes that the PRD never defines inline; correctness depends on the reader knowing the codebase.** The classification (L02/L06/L07 draft-tolerable; L03/L04/L05 always-enforced; L01 unchanged) matches the real findings in `crates/shirabe-validate/src/lifecycle.rs`, so it is accurate. But an implementer reading only the PRD cannot verify the split is exhaustive or correct without opening the source. Suggested fix: add a one-line gloss per code (e.g., "L02 = orphan/chain-connectivity, L03 = dependency cycle, ...") so the PRD is self-contained. The Functional section already glosses L02/L06/L07 and L03/L04/L05 in prose, but L01's behavior ("remains posture-sensitive as it is today") is asserted without describing what that current behavior is.

2. **The exit-code contract in R12 is incomplete versus the implementation.** R12 states "(0 clean, 2 violations, 1 tool-error)" but the actual CLI defines a fourth code, exit 3 (I/O error), as the most-severe outcome. The PRD's three-code summary is a subset. This matters because R13 forbids a network dependency in the gate path and the advisory layer reads "local, hermetic context" — an I/O failure reading that context could surface as exit 3, and the PRD never says whether advisory-layer I/O errors are allowed to change the verdict. Suggested fix: either acknowledge exit 3 in R12, or add a requirement that advisory-layer failures degrade gracefully (fall back to a verdict-only result) rather than escalating to a tool/IO error and changing the exit code.

3. **No acceptance criterion verifies R9's negative ("reading context to gate is forbidden").** R9 says ambient PR context MAY enrich phrasing but SHALL NOT change the verdict. The determinism AC (identical inputs/posture → identical verdict across environments) partially covers this, but does not directly test the case where PR context is *present and differs* (e.g., same documents and same declared posture but draft=true vs draft=false in the environment) yielding an identical exit code. Suggested fix: add an AC that varies ambient PR context while holding documents+posture constant and asserts the verdict (exit code + JSON) is unchanged.

4. **No acceptance criterion verifies R2's default-safety under an *explicit empty/unknown* posture signal.** The "no posture argument behaves as in-flight" AC covers the absent case. But R2 frames in-flight as the default "absent a positive review-ready signal" — it does not say what happens on a malformed or unrecognized posture value. Suggested fix: clarify whether an invalid posture argument is a tool-error (exit 1) or falls back to in-flight, and add an AC if behavior is defined. (May legitimately be deferred to DESIGN since the interface shape is out of scope; if so, state that explicitly.)

5. **The supersession of `DECISION-lifecycle-strict-mode-interface-2026-06-06` is recorded but has no acceptance criterion or migration note for the existing `--strict` boolean.** The current CI wires `--strict` from `github.event.pull_request.draft`. R3 says CI asserts review-ready "only when the pull request is ready-for-review," which implies the `--strict` plumbing is replaced or repurposed — but nothing in the ACs verifies the old `--strict` behavior still works (or is intentionally removed) for current consumers. R12 promises compatibility with `/scope` and cascade pass-throughs but those are pass-throughs, not the CI `--strict` caller. Suggested fix: add an AC or scope note covering what happens to the existing `--strict` flag and its single-pr re-targeting behavior (which R7 says L01 keeps).

## Suggested Improvements

1. **Name the maintainer-facing "single discoverable place" more concretely.** R10 and its AC require the classification be documented in "a single discoverable location," and the maintainer user story motivates it, but the PRD leaves the location fully open. A pointer (e.g., "a doc under docs/guides/ or a section the FC-CONVENTIONS check can locate") would reduce a guess at implementation time without over-constraining. Rationale: the maintainer story is the only one whose satisfaction is hard to verify without knowing where to look.

2. **Add a missing user type: the ad-hoc manual checker.** The Problem Statement explicitly lists four callers — local agent, CI-on-draft, CI-on-ready, and "ad-hoc manual checks" — but the User Stories cover only the first three (agent, draft-PR contributor, ready-PR author) plus the maintainer. The manual ad-hoc runner is folded into "agent drafting locally," which is reasonable, but a human running a one-off check against an already-merged or arbitrary doc set is a distinct scenario worth one sentence. Rationale: closes the gap between the four callers named in the problem and the stories that address them.

3. **Clarify L01's "posture-sensitive as it is today" against the new posture vocabulary.** The PRD introduces in-flight/review-ready posture, but L01's existing sensitivity is to the *single-pr-vs-multi-pr* posture and the `--strict` re-targeting — a different axis. Saying L01 "remains posture-sensitive" risks conflating the two posture concepts. Rationale: a reader could wrongly assume L01 now keys off draft/ready too.

## Summary

The PRD is well-structured and implementable: goals, stories, requirements, and acceptance criteria are tightly aligned, and its L0x classification matches the actual findings in the validate crate. The notable gaps are an incomplete exit-code contract (omits exit 3 / I/O error and never says how advisory-layer failures interact with the verdict), no direct AC for R9's anti-gating rule beyond determinism, and an unaddressed fate for the existing `--strict` boolean the supersession implies. None of these block implementation, but addressing them would remove the few remaining places an implementer must guess.
