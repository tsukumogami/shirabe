# Architecture Review
**Verdict:** PASS

The koto/plain hybrid, the E3/R1 extraction-plus-dispatcher, and the on-home-PR durable substrate together satisfy every PRD requirement (R1-R17) with parity-by-construction for the owned shapes, and each decision is grounded in real reference artifacts that I verified exist and say what the design claims.

## Issues Found

1. **(Minor, HOW-deferred) Coordinated carry-forward (R9) is asserted but not specified.** Decision 1's report explicitly flags this as an unresolved follow-on ("Whether carry-forward crosses repo boundaries in coordinated mode, and how, given F1 redaction"), yet the DESIGN's data-flow for coordinated (Solution Architecture step 2) lists "resolve gate nodes -> re-gate" with no carry-forward step, and the capability table only says the parent loop "assembles from child summaries." R9 says carry-forward is available in coordinated. The mechanism is genuinely under-specified for the coordinated branch — /plan will need to know whether the coordinated loop carries context forward at all, and if so from where. Not a blocker (single-pr inherits it verbatim; coordinated parity is the weaker of the two by the design's own admission), but it is the one HOW-gap /plan should be told to close.

2. **(Minor) Single-pr durable substrate is named two ways without disambiguation.** Decision 3 leaves single-pr's durable state as "PR body OR a tracked state file committed to the PR's branch and reachable via the PR ref" — an unresolved either/or. The DESIGN inherits the ambiguity ("the single PR for single-pr"). For coordinated the substrate is precise (PR-Index + merge-order DAG, recomputed live from gh). For single-pr, /plan needs to know whether resume reads PR body text or a committed ref'd file; these have different cross-branch-reachability and F4-style "never trust body text" implications. Bounded and resolvable at plan time, but currently HOW-missing.

3. **(Nit) Implementation Approach lists no explicit step for repointing /work-on's existing PLAN-detection evals.** Decision 2's "secondary risk" calls out that `skills/work-on/evals/evals.json` currently asserts plan-orchestrator behavior and must be repointed (single-pr/coordinated cases move to /execute's suite). The DESIGN's step 6 covers new backward-compat evals but does not mention retiring/repointing the old orchestrator evals in /work-on. Easy to fold into step 2, but worth flagging so the narrowing doesn't leave dangling failing evals.

## Strawman Check

PASS. The rejected options have genuine depth, not strawmen. Decision 1 Option A (single koto session) is rejected for a concrete, verified reason — koto has no cross-repo session, and the coordination contract's "no session file is the source of truth" (confirmed at coordination-strategy.md line 55) would make a plan-level session a competing authority; Option B is rejected because it discards the proven work-on-plan template and imposes real crash-resume regression risk, with the report acknowledging B is the *natural fit for coordinated* (the hybrid then adopts exactly that for coordinated — evidence the rejection is reasoned, not dismissive). Decision 2's E2 (shared library) is rejected on a substantive legibility argument plus the real `${CLAUDE_PLUGIN_ROOT}` triple-path maintenance cost; R2 (hard-remove PLAN input) is rejected for a concrete backward-compat break (existing invocations + the skill's own evals), and R3 (shared dispatcher) for genuine YAGNI over a stable three-value enum. Decision 3's option (a) wip-yaml-only is rejected against the actual, verified I-6 text (parent-skill-pattern.md lines 59-76, 217-225 confirm wip-yaml-md does NOT satisfy I-6); (c) koto-context-store is rejected as an unbuilt amplifier substrate but explicitly *kept as the named future alternate*, which is honest deferral rather than dismissal.

## Summary

The architecture is sound and PRD-complete. The koto/plain hybrid correctly uses koto exactly where single-repo nesting works (single-pr, inheriting all R8-R13 capabilities verbatim from the lifted work-on-plan template, which is parity-by-construction) and a plain durable-state loop where koto cannot reach (coordinated cross-repo), binding the done-signal to the verified `shirabe validate --merge-gate`; the E3 extraction plus R1 dispatcher keeps every existing PLAN doc and invocation running unchanged (backward-compat is structural — the per-issue child template stays byte-identical), and the on-home-PR durable substrate closes I-6 within v1 by augmenting resume-ladder rows 8-9 with a topic-keyed PR lookup, consistent with the actual resume-ladder template. The three open items are HOW-details for /plan (coordinated carry-forward mechanism, single-pr substrate disambiguation, old-eval repointing), none of which undermine the architecture.
