# Clarity Review

## Verdict: PASS
The PRD is specific enough that two developers would build substantially the same thing, but seven concrete ambiguities need resolution before implementation begins.

## Ambiguities Found

1. **R9 (`custom-statuses` input semantics)**: "extends or replaces the canonical enum for any format" -> The word "or" makes the merge behavior implementation-defined. One developer will union the override list with the canonical enum; another will replace it entirely. The example shows a full replacement (`[Draft, Accepted, Delivered, Done]`), but the prose says "extends or replaces." -> Specify the semantics explicitly: the override list replaces the canonical enum for that format in full. If extension is also wanted, add a separate `extra-statuses` input or document that callers must repeat canonical values they want to keep.

2. **R8 (filename prefix matching — case sensitivity and path depth)**: "filename prefix: `DESIGN-`, `PRD-`, `VISION-`, `ROADMAP-`, `PLAN-`" -> It's unclear whether the prefix match is case-sensitive, and whether the match is on the filename alone or the full path. A file at `docs/designs/design-foo.md` (lowercase) or `src/DESIGN-bar.md` (no designated directory) would behave differently depending on the implementation. -> State explicitly: prefix matching is on the basename only, is case-sensitive, and matches only uppercase prefixes as listed.

3. **R7 (public repo VISION check — CI behavior on private repos and on forks)**: "when the workflow runs in a public repository" -> `github.repository_visibility` is `public` for public repos, but forks of public repos are also `public`. A fork's PR that adds `## Competitive Positioning` would fail, which may be surprising. Additionally, the visibility context variable (`github.repository_visibility`) is not always set in all GHA contexts. -> Clarify: (a) confirm that forks of public repos should also have this check applied; (b) specify the fallback behavior if `github.repository_visibility` is unset or empty (fail closed / skip the check / treat as public).

4. **R6 (Plan structural rules — "tracked by git" definition)**: "the referenced file must exist on disk and be tracked by git" -> "Tracked by git" is ambiguous in a PR context: does it mean tracked in the base branch, tracked in the PR branch (staged), or simply present on disk? A new upstream Design doc added in the same PR as a Plan doc that references it may not yet be in `git ls-files` depending on when the check runs. -> Define the check precisely: the upstream file must appear in `git ls-files HEAD` on the PR branch at the time the workflow runs, which includes files staged and committed in the PR.

5. **R11 (GHA annotations — line number for frontmatter errors)**: "emitted as GitHub Actions error annotations (`::error file=<path>,line=<N>::<message>`)" -> FC01 (missing field), FC02 (bad status value), and FC03 (frontmatter/body mismatch) are frontmatter errors. The line number `<N>` is unspecified: is it the line of the failing field, the first line of the frontmatter block, or line 1 of the file? Different implementations will produce different annotation positions. -> Specify: for FC01 (missing field), annotate line 1 (field is absent, no line to point to); for FC02, annotate the line containing the `status:` key; for FC03, annotate the `## Status` heading line in the doc body; for FC04, annotate line 1 of the file.

6. **R15 (fast execution — "typical PR touching 1-5 doc files")**: "completes in under 60 seconds for a typical PR touching 1-5 doc files" -> "Typical" is undefined. The acceptance criterion (`under 60 seconds for a PR touching 5 doc files`) is more concrete, but "typical" still leaves room for a developer to argue that a 5-file PR is not typical. -> Replace "typical" with the specific bound from the acceptance criteria: "under 60 seconds for a PR touching up to 5 doc files."

7. **R16 (branch protection update — "replacing or supplementing")**: "replacing or supplementing the existing `check-plan-docs` workflow" -> The word "or" leaves the action undefined. The existing `check-plan-docs` job is a required status check; removing it without confirming `validate-docs` covers the same ground could silently drop protection. Adding both creates redundant checks. -> Make the decision: if `validate-docs` subsumes all checks from `check-plan-docs` (which R6 implies), then `check-plan-docs` should be removed. State that explicitly as the intended action.

## Suggested Improvements

1. **Add an acceptance criterion for `custom-statuses` replace semantics**: The existing criterion only tests that `Delivered` is accepted. Add a criterion that a PRD with `status: Draft` is also accepted when `prd: [Accepted, Delivered, Done]` is passed (i.e., Draft is no longer valid after a full replacement). This makes the replace-vs-extend question testable.

2. **Specify the diff base for changed-files detection**: R2 says `git diff --name-only BASE...HEAD` without defining `BASE`. In a `pull_request` event, `BASE` is typically `github.event.pull_request.base.sha`, but in other trigger contexts (push, `workflow_dispatch`) it could be undefined. Document what `BASE` resolves to and what happens if the workflow is called from a non-PR context.

3. **Define the error message format for FC03**: The acceptance criterion for FC03 says the job must fail with "an FC03 annotation," but doesn't specify what the message text must contain. US2 says contributors should be able to fix errors "without reading the format spec." A message like `FC03: frontmatter status 'Accepted' does not match body status 'Draft'` satisfies that intent; a bare `FC03 mismatch` does not. Adding a required message template (even loosely specified) closes the gap.

4. **Clarify version pinning behavior for the `v1` tag**: The acceptance criterion says "downstream repos pinning `@v1` receive all patch improvements automatically." This relies on a floating `v1` tag (a mutable ref updated on each patch release). This is a convention, not enforced by GHA. Confirm in the PRD — or in a Known Limitations entry — that shirabe will use a mutable `v1` tag following the same pattern as `actions/checkout@v4`.

## Summary

The PRD is well-structured and mostly implementation-ready. The problem, goals, and acceptance criteria are concrete and largely binary. The seven ambiguities cluster around three areas: the `custom-statuses` merge semantics (R9), the precision of error annotation line numbers (R11), and several branch conditions that are documented as "or" when only one outcome is intended (R6, R7, R16). None of these is a fundamental design gap — each has an obvious resolution — but leaving them open will produce divergent implementations or post-review debates. Resolving them before the implementation issue is filed will prevent rework.
