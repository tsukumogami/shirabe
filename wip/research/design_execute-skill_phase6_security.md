# Security Review

**Verdict:** PASS

The design names all six parent-skill security surfaces, keeps untrusted PLAN content as data, and binds the fail-closed merge-gate correctly; the gaps are completeness-of-prose, not load-bearing holes, and belong in the implementation tickets.

## Findings

1. **Slug re-validation on resume**: ADDRESSED in intent, UNDERSPECIFIED in mechanics. The Security Considerations list names "Slug re-validation on resume" and the resume ladder does a "topic-keyed home-PR lookup (via `gh`) before declaring no state." But /execute introduces a recovery source the reference does not cover: the slug is recovered from a live `gh` PR lookup (home-PR by topic), not only from a Slot-5/6 on-disk artifact path. The `^[a-z0-9-]+$` re-validation MUST also apply to the slug as keyed into and recovered from that `gh`-backed lookup before it is interpolated into any `gh` argv or state-file write path. -> Implementation must state explicitly that the topic-slug regex re-gates the home-PR-recovered slug, not just the file-glob-recovered one. Low risk because `gh` lookups are read-only and the slug is operator-supplied, but the path-traversal/injection surface on the subsequent write/branch interpolation is real.

2. **Closed write-target set**: ADDRESSED. The design enumerates writes: `wip/execute_<topic>_state.md` scratch, the durable home PR (its own body, in its own repo), and `run-cascade.sh`'s upstream-lifecycle transitions (PLAN removal + BRIEF/PRD/DESIGN/ROADMAP). This matches the pattern's enumerable-set rule and the R9 hard-finalization check is cited. -> SKILL.md must literally enumerate these so R9 can enforce membership; flag for the implementation ticket, not a design blocker.

3. **Enum re-validation before path interpolation**: ADDRESSED. `execution_mode` re-validation "before any path/branch interpolation" is named explicitly. This is the load-bearing one for /execute because `execution_mode` selects the branch (koto single-pr vs plain coordinated loop) AND feeds `impl/<slug>` branch construction. The design correctly places re-validation ahead of branch/path use. -> No fix; ensure the dispatcher in /work-on re-validates too (it reads `execution_mode` to route), since it is a second consumer of the same untrusted enum.

4. **Stale `parent_orchestration:` self-heal**: ADDRESSED. The `parent_orchestration:` sentinel is carried in the wip-yaml-md projection and named as a security surface. -> The reference requires an UNCONDITIONAL silent clear at Phase 0 (no prompt, no warning, no recency check). The design does not restate the unconditional-clear semantics; implementation must, because a conditional self-heal is a conformance break. Flag, not a blocker.

5. **Visibility boundary**: ADDRESSED, and correctly delegated. /execute is declared a public skill; coordinated mode inherits the F1/Coordination-PR Visibility Rule from coordination-strategy.md (most-restrictive visibility, public PR never indexes a private repo, fail-closed-to-private on unresolvable). The design consumes a /scope-created coordination home rather than authoring cross-visibility references itself, which keeps it on the safe side of the Public->Private directional rule. -> One gap: the design says /execute "consumes" the coordination PR but re-authors/refreshes the body during Track (per coordination-strategy lifecycle phase 2). When /execute re-authors the body it becomes an authoring consumer of F1/F2/F3, so its SKILL.md MUST carry the F1 redaction + F2 `owner/repo:path` component validation + F3 control-char stripping bindings, not just F4. The design only foregrounds F4 (`--merge-gate`). Surface F1/F2/F3 in the coordinated-path prose.

6. **No untrusted-input interpolation**: ADDRESSED, and this is the strongest part. The design states plainly "no interpolation of untrusted PLAN-body content into emitted shell commands (PLAN content is data, not instructions)." Children (`work-on.md`) own author-input interpolation via their own stdin discipline; /execute's read surface is metadata-only (issue/PR/unit status, terminal states, summaries). This matches the pattern's metadata-read assignment. -> No fix.

7. **NEW cross-skill koto-template coupling (path traversal / silent break)**: ADEQUATELY MITIGATED for silent-break; PARTIALLY for traversal. The coupling is `koto init` against `${CLAUDE_PLUGIN_ROOT}/skills/work-on/koto-templates/work-on.md`. The path is a fixed literal anchored to the harness-provided `CLAUDE_PLUGIN_ROOT` plus a constant relative tail with no untrusted interpolation — a malicious PLAN slug cannot redirect it because the slug is never a component of this path (it goes into `impl/<slug>` branch and `wip/execute_<slug>_state.md`, both separately regex-gated per finding 1/3). So a crafted PLAN cannot redirect the template load. The residual risk is `CLAUDE_PLUGIN_ROOT` itself being attacker-controlled, which is a harness-trust assumption shared by every skill and out of scope here. The named mitigation (end-to-end eval asserting the resolved path + guarded preflight check) covers the silent-break failure mode correctly. -> Preflight check should assert the resolved path is a regular file under the expected plugin root (reject symlink-escape), making the traversal argument explicit rather than implied.

8. **Fail-closed merge-gate done-signal**: ADDRESSED, correctly. The design binds the done-signal to `shirabe validate --merge-gate --mode=ready` recomputed from live `gh`, with "a `gh` failure halts rather than falsely signaling done" and "`gh` auth is a precondition." This matches F4 (recompute from live `gh`, never PR-body text; any unresolvable PR treated as not-merged; pinned to `draft == false`). A gh failure -> halt -> never falsely-done is the genuine fail-closed posture. -> No fix.

9. **Finalization cascade atomicity**: ADDRESSED. `run-cascade.sh` is described as atomic and self-verifying via pre/post probes so a partial finalization cannot silently leave the upstream chain inconsistent. The design reuses the proven cascade rather than reimplementing it, so the atomicity property is inherited, not newly asserted. -> One concern to flag for implementation: in coordinated mode finalization is distributed (each repo finalizes in its own PR, the coordination PR merges last as the all-or-nothing gate). "Atomic" there means the merge-gate is the single commit point, not that the cascade script is transactional across repos. The design's claim is consistent with coordination-strategy's read-only cross-repo gate, but the SKILL.md should not overstate cascade atomicity as spanning repos — the gate, not the script, is the coordinated done-point.

10. **Additional gap — dispatcher as a second untrusted-enum consumer**: The /work-on thin dispatcher reads `execution_mode` to route (multi-pr in place vs hand-off). This is a second trust boundary the Security Considerations section does not call out. It must re-validate `execution_mode` against its enum before branching, same as /execute, since a tampered PLAN reaching /work-on directly bypasses /execute's check. -> Add the dispatcher to the enum-re-validation surface explicitly.

## Summary

The design is sound and passes the mandatory security pass: all six parent-skill surfaces are named, untrusted PLAN content is kept strictly as data, and the merge-gate done-signal is genuinely fail-closed against live `gh`. The new cross-skill koto-template coupling does not open a slug-driven redirection because the template path is a fixed `CLAUDE_PLUGIN_ROOT`-anchored literal with no untrusted interpolation, and the named preflight+eval mitigation covers the silent-break mode. The findings are completeness-of-prose obligations for the implementation tickets — restate the unconditional `parent_orchestration:` clear, extend slug re-validation to the gh-recovered slug, surface F1/F2/F3 (not only F4) in the coordinated re-authoring path, and add the /work-on dispatcher to the enum-re-validation surface — none of which are load-bearing holes that should block the design.
