# Testability Review

## Verdict: PASS

A test plan could be written from the 27 acceptance criteria alone. Each AC names a concrete trigger, a specific observable outcome (file existence, file path patterns, frontmatter field values, message wording, HTML-comment markers, validator pass/fail), and the requirement it traces back to. A small number of criteria carry caveats around subjective signals or design-team-scope mechanisms, but in each case the AC explicitly pivots to an observable outcome. The PRD passes testability.

## Untestable Criteria

1. **AC6 (thesis-shift surfaces in Phase 1 discovery → chain includes `/vision`)**: "Thesis-shift signal" is described in R4 as author-stated and surfaced conversationally — there is no fixed input shape that produces this signal deterministically. -> Make testable by either (a) defining a fixture set of "thesis-shift utterance examples" in the eval corpus that the eval scenarios use as test inputs, or (b) requiring the SKILL.md Phase 1 prose to list 2-3 concrete signal exemplars that a test plan can replay verbatim. The AC can then check "given utterance X from the fixture, chain proposal includes `/vision`."

2. **AC20 (re-entry not hijacked by child's "offer to revise" prompt)**: The AC explicitly defers the mechanism ("signalling mechanism is design-team scope") and asks the AC to "verify observable outcome." The observable outcome is named only as "`/charter` flow not hijacked" — which is itself the unobservable thing. -> Tighten to a specific observable: e.g., "When `/charter` invokes `/strategy` with an Accepted STRATEGY on disk, the next prompt the author sees is one of {Re-evaluate, Revise, Bail} from `/charter`'s prompt vocabulary — NOT `/strategy`'s '/Continue / Start fresh' vocabulary." That makes "hijack" concrete via prompt-string match.

3. **AC27 (pattern-level requirements reflected in downstream design doc)**: The AC is "post-design" — it verifies a not-yet-existent design doc. There's no test plan that can run today; the criterion is a future cross-document check. -> Either (a) move AC27 to a tracking checkbox outside the AC list (it's a meta-commitment, not a property of `/charter` itself), or (b) define the check operationally: "Every requirement tagged `[pattern-level]` in this PRD must appear as a section heading or labeled requirement in DESIGN-shirabe-progression-authoring.md when that doc reaches Draft." Then the test plan owner can grep the design doc once it exists.

4. **AC18 wording check ("Re-evaluate / Revise / Bail" — NOT "Continue / Start fresh")**: This is testable as a prompt-string match, but the AC is silent on whether the wording must be verbatim or substring-match. -> Specify: "The prompt MUST contain the literal substrings 'Re-evaluate', 'Revise', and 'Bail' (case-insensitive), and MUST NOT contain the literal substring 'Continue / Start fresh'." Verbatim vs substring matters for eval automation.

## Missing Test Coverage

1. **R1 SKILL.md template shape compliance**: R1 enumerates 7 required SKILL.md sections (input modes, execution-mode flag parsing, topic-slug constraint, Workflow Phases diagram, Resume Logic ladder, Phase Execution list, Reference Files table). AC1 only checks loadability and frontmatter `name: charter`. No AC verifies the 7 sections are present and non-empty. -> Add AC: "skills/charter/SKILL.md contains sections matching each of the 7 required structural elements named in R1."

2. **R6 STRATEGY-path rejection**: R6 states `/charter` MUST NOT pass a STRATEGY path to `/strategy`. No AC verifies this negative case. -> Add AC: "When the chain has an existing STRATEGY for the topic and is re-running via /charter, /charter passes /strategy either a topic slug, VISION path, or PRD path — never the STRATEGY path."

3. **R6 input variants to `/strategy`**: R6 names three valid input shapes to `/strategy` (freeform topic, VISION path, PRD path). ACs only implicitly cover the freeform/VISION cases. No AC covers "chain operationalizes a feature PRD." -> Add AC: "When `/charter` Phase 1 identifies an existing PRD as the chain's framing, `/strategy` is invoked with the PRD path as upstream."

4. **R7 `/roadmap` upstream-rejection caveat**: R7 says "If `/roadmap`'s phase code rejects a STRATEGY path as upstream, the delegation contract MUST surface the gap explicitly — `/charter` SHALL NOT silently substitute a VISION path." This is a failure-mode requirement. No AC covers it. -> Add AC: "When `/roadmap` rejects the STRATEGY path passed as `--upstream`, `/charter` surfaces a visible error to the author rather than silently downgrading to VISION upstream."

5. **R14 child-internals isolation**: R14 forbids the parent from inspecting child `wip/research/<child>_<topic>_phase<N>_*.md` files. No AC verifies this isolation. -> Add AC: "During and after a `/charter` chain, `/charter`'s decision logic depends only on (a) child doc frontmatter, (b) the topic slug, and (c) its own state file — never on the child's internal `wip/research/` files." (Testable via mock/spy on file reads or via code review against the SKILL.md prose.)

6. **R17a/R17b CLAUDE.md surfacing**: AC list has no check that CLAUDE.md updates ship with `/charter`. -> Add AC: "Workspace and shirabe CLAUDE.md files mention `/charter` and include the trigger phrases listed in R17b."

7. **R18 evals scenarios cover US-1..US-4**: AC list has no check that `evals.json` ships with scenarios covering all four user stories. -> Add AC: "`skills/charter/evals/evals.json` contains at least one scenario named or tagged for each of US-1, US-2, US-3a, US-3b, US-4, and all scenarios pass under `scripts/run-evals.sh charter`."

8. **R8 STRATEGY-only full-run exit (no ROADMAP)**: AC11 is the full-run AC but reads "Draft STRATEGY (and optional Draft ROADMAP)" without separating the two sub-cases. The AC passes if either case ships. -> Split into AC11a (STRATEGY-only full-run) and AC11b (STRATEGY + ROADMAP full-run). The two are functionally different exits per R7 and deserve independent verification.

9. **Edge case: mid-chain `/vision` rejection or `/comp` failure**: ACs cover `/strategy` rejection (US-3a) but don't address what happens if `/vision` is invoked and the author bails inside `/vision` (or `/comp` errors out). The chain has multiple invocation points; only one of them has a tested rejection path. -> Add AC: "When the author bails inside an invoked child (`/vision` or `/comp` or `/roadmap`), `/charter`'s resume ladder on next entry routes to abandonment-forced exit per US-3b semantics."

10. **Edge case: topic-slug derivation rules**: AC3 covers slug rejection on whitespace, but R3's `^[a-z0-9-]+$` regex also bans uppercase, underscores, and special characters. AC3 tests one violation only. -> Add AC: "`/charter` rejects `$ARGUMENTS` containing uppercase letters, underscores, or special characters at Phase 0 with a clear error naming the violated pattern."

11. **Edge case: state file corruption / missing fields on resume**: R10 specifies many state-file fields. No AC tests behavior when the state file exists but is malformed or missing required fields (`exit:` without `decision_record_sub_shape:` when it should have one, etc.). AC15 covers the finalization-time check; nothing covers the resume-time check. -> Add AC: "On resume, when `wip/charter_<topic>_state.md` is malformed or missing required fields for its recorded phase, `/charter` surfaces a clear error and offers Discard as a recovery path."

12. **R15 Decision Record validator coverage**: AC25 names the required ADR-style sections but doesn't check the per-sub-shape body content R15 enumerates (re-evaluation Context cites new evidence; rejection Options Considered names "accept the Draft" and "revise instead of reject"). -> Add AC: "Re-evaluation Decision Record's Context section cites at least one new evidence item, and Options Considered lists 'revise the STRATEGY' and 'force-abandon and rewrite' as rejected alternatives" + analogous AC for rejection sub-shape.

## Suggested Improvements

1. **Adopt a Given/When/Then template for ACs**: Several ACs combine the precondition, the action, and the observable outcome in one sentence (e.g., AC10, AC13). Splitting into Given/When/Then makes the test plan author's job mechanical. Rationale: the PRD currently leaves "what's the setup" implicit; a test plan author has to re-derive the precondition (does the topic already have a STRATEGY? does CLAUDE.md declare visibility? what state file exists?). Explicit preconditions remove ambiguity.

2. **Add a "Test Fixtures" section to the PRD**: Specify the inputs the test plan needs to be deterministic — example CLAUDE.md visibility headers (Public/Private/missing), example existing-STRATEGY frontmatter, example author utterances that should trip thesis-shift signal vs not. Rationale: addresses untestable-criteria #1 (thesis-shift signal) and gives the test plan author concrete fixtures rather than asking them to invent canonical examples.

3. **Distinguish "feature ACs" from "meta-commitment checks"**: AC27 (pattern-level reflected in downstream design doc) and AC18 (load-bearing wording from US-2) are different kinds of checks than AC1-AC17. The first verifies a separate artifact's content; the second verifies a UX commitment the brief makes load-bearing. Separating them into a "Meta-commitments" subsection clarifies which ACs run against `/charter` source and which run against external artifacts. Rationale: keeps the test plan scoped to `/charter`'s own surface, with cross-doc checks called out as separate verification steps that may not be runnable until the cross-doc lands.

4. **Tag each AC with its automation level**: Mark each AC with one of [automated-unit, automated-eval, manual-review]. AC1, AC3, AC4, AC15, AC24, AC25, AC26 are mechanically checkable; AC6, AC18, AC20 require eval-style scenarios or manual review of prompt wording; AC27 is post-hoc manual. Rationale: makes the eval/manual split visible up front; helps R18 evals.json scope.

5. **Lift the `[Repo Visibility]` reading dependency into an explicit test prerequisite**: R12 and AC21 both depend on parsing CLAUDE.md's `## Repo Visibility:` header. A test plan needs to manipulate this file; calling that out (and noting fixture #2 above) keeps the test plan reproducible across machines. Rationale: avoids the test plan accidentally depending on the actual repo's CLAUDE.md.

6. **Cross-link ACs to user stories, not just requirements**: ACs trace to requirements (R1-R18). They don't trace to user stories (US-1..US-4). Adding a US column would make it possible to ask "is US-2 fully covered?" Rationale: requirement-level coverage doesn't guarantee story-level coverage; AC18 is explicitly tied to US-2 wording but no other ACs are story-tagged.

## Summary

The PRD's 27 ACs cover the core mechanics — slug validation, child invocation gating, exit-path enforcement, state-file finalization, resume ladder, manual-fallback isolation, and schema compliance — concretely enough that a test plan author could write coverage from the criteria alone. Three ACs slide toward subjective territory (AC6 thesis-shift, AC20 child re-entry hijacking, AC27 cross-doc reflection) and need either fixtures, observable-prompt-substring contracts, or relocation outside the AC list. About a dozen test-coverage gaps exist — primarily around negative cases (STRATEGY-path rejection in R6, `/roadmap` upstream-rejection in R7), child-internals isolation (R14), CLAUDE.md surfacing (R17), eval scenarios (R18), and edge cases on slug derivation and state-file corruption. None of these gaps undermine the testability verdict on their own; the PRD's structural decision to bind each AC to a requirement ID and to specify observable outputs (file paths, frontmatter field values, prompt substrings) earns it a PASS, with the test-coverage gaps flagged as additions for the test plan author to land before implementation.
