# Completeness Review

## Verdict: PASS

The PRD covers the brief's four User Journeys with five user stories (US-3 split into US-3a/US-3b to honor the rejection sub-shape decision), and the requirements are dense enough that an implementer can build `/charter` without guessing. The gaps below are improvements, not blockers: the load-bearing contracts (R8 three exits, R10 state schema, R11 resume ladder, R7 `/roadmap` handoff, R4-R6 child invocation signals) all have matching acceptance criteria, and the reference paths cited in the PRD resolve to real files where they should — `skills/comp/` is correctly acknowledged as unshipped, and `docs/decisions/` is expected to be created by `/charter` itself.

## Issues Found

1. **Entry-router "Revise" path has no acceptance criterion.** US-2 (the load-bearing story) names three entry-router options ("Re-evaluate / Revise / Bail") that fire when an Accepted/Active STRATEGY exists. AC12 covers Re-evaluate, AC18 verifies the wording, but no AC verifies what happens when the author picks **Revise** — does `/charter` invoke `/strategy` against the existing STRATEGY path (Input Mode 2 lifecycle verb? Input Mode 3 fresh draft?), and does the resulting chain land at full-run with the revised Draft? Suggested fix: add an AC under "Exit-path enforcement" stating that "Revise" at the entry router invokes `/strategy` (specify which input mode), produces a revised Draft STRATEGY, and the chain exits at full-run with the revised artifact recorded in `exit_artifacts`.

2. **"Bail" routing at the entry router lacks an explicit AC.** R8 and Decision 3 commit to "no true bail path; bail routes to abandonment-forced." US-2's entry router offers Bail as one of three options. No AC verifies that selecting Bail at the US-2 entry router routes to abandonment-forced when prior wip state exists (and to a clean cancel when no wip state exists, per the user-researcher research). Suggested fix: add an AC explicitly verifying the Bail branch behavior at the entry router and at the chain-proposal confirmation prompt.

3. **The `_discover.md` vs `_scope.md` asymmetry has no AC.** R11 documents that `/strategy` writes `wip/strategy_<topic>_discover.md` but its SKILL.md ladder references `_scope.md`; `/charter`'s ladder MUST read `_discover.md`. This is called out as a known asymmetry but no AC verifies that `/charter`'s resume ladder reads `_discover.md` (not `_scope.md`) when detecting a partial `/strategy` run. Without an AC, a future change to `/charter` could silently regress to reading `_scope.md` and the resume ladder would fail to fire on partial `/strategy` state. Suggested fix: add an AC under "Resume ladder" stating that when `wip/strategy_<topic>_discover.md` exists (the file actually written by `/strategy` Phase 1) but `wip/strategy_<topic>_scope.md` does not, `/charter` detects the partial `/strategy` run and resumes into `/strategy`.

4. **Stale-session "Force-materialize" choice path has no end-to-end AC.** AC17 verifies the three-option prompt (Resume / Force-materialize / Discard) fires at ≥7d. AC14 verifies the abandonment-forced artifact shape. But no AC ties the two together: when the author at AC17 selects "Force-materialize," does `/charter` execute the force-materialization and write the HTML-comment marker per AC14? Suggested fix: chain AC17 → AC14 explicitly, or add an AC stating that Force-materialize selection at the stale-session prompt triggers the abandonment-forced exit (writes the marker, sets state-file fields).

5. **Chain proposal confirmation prompt is not specified at requirement altitude.** US-1 narratively describes a chain proposal at the end of Phase 1 discovery ("here's the chain I propose: skip `/vision`, run `/strategy`, run `/roadmap` — Proceed / Adjust chain / Bail"). The user-researcher research surfaces three such confirmation prompts (after Phase 1, after `/strategy`, after `/roadmap`). No requirement codifies that `/charter` MUST surface a chain-proposal confirmation, what its three options are, or what "Adjust chain" lets the author do. Suggested fix: add a functional requirement (`[/charter-specific]`) stating that `/charter` Phase 1 SHALL conclude with a chain-proposal confirmation prompt offering Proceed / Adjust / Bail, and define what "Adjust" enables (re-route the chain shape before any child fires).

6. **AC27 is unfalsifiable at PRD acceptance time.** "Each requirement tagged `[pattern-level]` in this PRD is reflected in `docs/designs/DESIGN-shirabe-progression-authoring.md`'s pattern-level scope" is checkable only after the design exists. As a binary AC at PRD acceptance it is vacuous: there is no design doc to check against. Suggested fix: either reframe AC27 as a contract-passing-the-baton commitment (the PRD's pattern-level tags are present and consistent, verifiable now) and move the post-design check to the design's own acceptance criteria, OR drop AC27 from the PRD's AC list and surface it as an Open Question / Downstream Artifact dependency.

7. **No requirement covers what happens when `/strategy` Reject fires *outside* a `/charter` chain.** R13 and US-4 cover the manual fallback path for revising a STRATEGY, but the manual-fallback equivalent of US-3a (author invokes `/strategy` directly, runs to Phase 5, picks Reject) is silent in the PRD. Should `/charter` retroactively write a Decision Record when the next `/charter` resume detects the discard commit? Or is the rejection-sub-shape Decision Record exclusively a `/charter`-orchestrated artifact? Suggested fix: add a sentence (likely in R8 or in Decision 5's reasoning) stating that the rejection sub-shape Decision Record is `/charter`-orchestrated only — manual `/strategy` Reject outside `/charter` leaves only the discard commit as the durable trace, by design. This closes a scope question that a future reader will otherwise re-derive.

8. **R10 state-file schema mixes optional and required fields without explicit nullability.** The state-file schema lists fields like `decision_record_sub_shape` ("set ONLY when exit=re-evaluation") and `discard_commit_sha` ("set on decision-record/rejection") as conditional, but does not state whether they MUST be absent or MUST be set to null/empty in other cases. R9 only checks that `decision_record_sub_shape` is set when `exit: re-evaluation`; it doesn't verify these fields are absent or null otherwise. Suggested fix: add to R9 (or R10) that conditional fields MUST be unset (absent from the frontmatter) when their triggering condition does not hold, OR explicitly state that extra fields are tolerated.

## Suggested Improvements

1. **Add an AC for the visibility-default warning text.** R12 mandates a warning when CLAUDE.md lacks `## Repo Visibility:`. AC21 verifies the default-to-Private behavior and that a warning fires "naming the missing header." Strengthening: spell out the warning text in R12 (or in a known-good rubric) so the warning isn't reinvented per implementer. Rationale: the inherited pattern from `/strategy` already has working warning text; cite the exact phrasing rather than re-deriving.

2. **Tighten Open Question 1 into a Phase-0 design dependency.** OQ1 ("`/roadmap`'s `--upstream` STRATEGY-path acceptance") is named as pending verification but R7 already encodes a delegation contract that fails if the verification fails. The PRD could surface this dependency more sharply by stating that the design phase MUST resolve OQ1 before `/charter`'s Phase 0 implementation locks; otherwise R7's "MUST NOT silently substitute a VISION path" provision becomes dead text. Rationale: improves traceability between OQ resolution and implementation gating.

3. **Cross-reference US-1 to AC11 explicitly.** US-1 names the full-run end state; AC11 verifies the state file but doesn't explicitly state which user story it traces back to. Adding parenthetical story-references to ACs (similar to the parenthetical requirement-references already present) would help a reviewer audit story-to-AC coverage. Rationale: light editorial improvement; the coverage is present, the traceability is implicit.

4. **Clarify R11's child-doc inspection scope for `/roadmap`.** R11 lists `child_snapshots` entries for `vision`, `strategy`, and `roadmap`, but the resume ladder ordering shows STRATEGY status checks but no ROADMAP status checks. If a Draft ROADMAP exists from a prior chain (full-run that landed both Draft STRATEGY and Draft ROADMAP), and the author re-invokes `/charter`, what happens? Suggested clarification: add a ROADMAP status case to the resume ladder, or state explicitly that `/charter`'s resume reasons over STRATEGY status as the chain anchor and ROADMAP follows the staleness-detection path of US-4.

5. **Bridge R18 evals to specific scenario shapes.** R18 says "eval scenarios MUST cover the four user stories (US-1 through US-4)." Five user stories now exist (US-3a and US-3b). Suggested fix: rephrase as "MUST cover US-1, US-2, US-3a, US-3b, and US-4" or simply "all user stories defined in this PRD." Rationale: aligns the count with the user-stories list.

6. **Out of Scope: add an explicit exclusion for cross-topic resume.** Open Question 5 flags cross-branch state-file behavior. Out of Scope is silent on whether `/charter` can resume against a *different* topic in the same session (e.g., author runs `/charter foo`, bails, starts `/charter bar`). The state-file is topic-keyed by filename so the answer is "they don't conflict" but stating this explicitly forecloses a class of bug reports. Rationale: defensive scope clarity.

## Summary

The PRD is publication-ready as a Draft: requirements R1-R18 cover the parent-skill template, the four delegation contracts, the three exit paths (with re-evaluation's two sub-shapes well-distinguished), the hybrid resume ladder, visibility gating, and the manual-fallback first-class contract. Reference paths to skills and docs resolve (the missing `skills/comp/` and `docs/decisions/` paths are correctly framed as future state, not broken references). The seven issues above are gap-fills — they tighten the AC coverage of the entry router's Revise/Bail branches, formalize the chain-proposal confirmation prompt, and remove one unfalsifiable AC — but none of them invalidates the requirements as written.
