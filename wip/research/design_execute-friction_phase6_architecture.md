# Phase 6 Architecture Review — DESIGN-execute-friction

VERDICT: PASS

## Findings

- **R→decision coverage (PASS):** All eight requirements map cleanly.
  - R1 → D1 (mode-aware: single-pr adopt + coordinated per-repo worktree / coordination-branch-docs-only). Both halves of R1's mode-aware mandate are addressed.
  - R2 → D2 (pause-as-suspension at the `pr_finalization`→`plan_completion` boundary).
  - R3 → D3 (`/plan`-owned docs emit + `user_visible_surface` signal + review-plan backstop).
  - R4 → D6 (fold template authoring into `pr_finalization`).
  - R5 → D4 (reuse `validate --lifecycle-chain --mode=ready`).
  - R6 → D5 (GitHub-issue convention + `docs/` fallback).
  - R7 (parity) → addressed in D1 via the `|| impl/$PLAN_SLUG` fallback and the create-path settled-branch = `impl/<slug>` invariant; Decision Outcome item 1 and Consequences both assert byte-for-byte parity.
  - R8 (autonomy) → addressed in D2 by binding pause to interactive mode so `--auto` is structurally incapable of pausing.
  No unaddressed requirement.

- **Considered Options depth (PASS):** Each decision presents genuine alternatives with honest, source-grounded rejections, not strawmen. D1 carries four options (mode-agnostic adopt, explicit flag, link-don't-adopt) each rejected for a specific reason traceable to the decision-1 report (split durable home / orphaned-second-PR / coordination-branch-must-stay-docs-only). D4 rejects a new mode and a renderer subcommand citing the CLAUDE.md CLI-surface contract and the removed `shirabe coordination` precedent. D6 rejects `/fix-pr` for the real reasons (second remediation pass violating R4, runtime cross-plugin coupling). Outcomes follow from the options.

- **D1 soundness (PASS):** Capturing the settled `HEAD` branch and injecting it into both `spawn_and_await` `jq` ticks (replacing the hardcoded `impl/$PLAN_SLUG`) is the minimal correct fix — the decision-1 report confirms the child treats `SHARED_BRANCH` as an opaque string and re-derives nothing, so no `/work-on` change is needed. The mode-aware split is coherent: single-pr adopts the home PR (correct, since `/execute`'s durable source of truth IS the home PR and a second linked PR would break topic-keyed resume); coordinated keeps code in per-repo worktrees and the coordination branch docs-only (matches existing coordinated dispatch). R7 parity holds via the `|| impl/$PLAN_SLUG` fallback plus the create-path recording `impl/<slug>` as the settled branch.

- **D2 soundness (PASS, with one prose inconsistency — see below):** Pause-as-suspension is consistent with the koto state machine. The report establishes koto has no human-wait primitive, so a terminal is the correct encoding of a resumable stop; routing a guarded second edge out of `pr_finalization` to a non-failure `paused_for_review` terminal reuses the exact seam #117's DRAFT-before-READY reorder cut (cascade + `gh pr ready` welded together in `plan_completion`, `pr_finalization` left as body-assembly-only). The pause sits one state UPSTREAM of the welded pair, so it never splits the weld and there is no half-finalized-chain risk. The `exit:` UNSET + resumable marker, with R9 firing only at terminal exits, is the decision-2 report's "preferred" encoding and is internally consistent. Interactive-vs-auto as the governor is actually a STRONGER choice than the report's recommended `--pause-for-review` flag: a flag orthogonal to `--auto` would make `--auto --pause-for-review` representable, which R8 forbids; binding pause to interactive mode makes that illegal combination unrepresentable by construction. Good architectural call.

- **D3 soundness (PASS):** `/plan` is the right owner — it is the only layer that reads the DESIGN body where the signal lives and the only layer that emits the issue set, so a docs item can be added before the run ever reaches a done-signal. A check inside `/execute` would require reading child bodies, violating the metadata-only contract (R14/R15) — correctly rejected. The `user_visible_surface` structured flag (authoritative) + `docs/guides/*` prose fallback (back-compat) + review-plan Scope-Gate backstop is a coherent generate-then-review layering. The flag-is-deterministic / prose-is-brittle reasoning is sound.

- **D4 soundness (PASS):** Reusing `validate --lifecycle-chain --mode=ready` does not conflate concerns — "is this chain finalized?" is exactly the ready-posture L01 check the cascade already self-verifies with, so promoting it to the R5 guard keeps a single implementation of "finalized terminal" shared between cascade self-check and human/CI guard. The seed-doc constraint is handled honestly (Decision Outcome item 4: post-finalization seed on the durable DESIGN anchor since the PLAN is gone), matching the decision-4 report's seed-selection rule. The draft-gated CI wiring (`pull_request.draft == false`) prevents false-fire mid-flight.

- **D5/D6 soundness (PASS):** D5 is correctly framed as a convention gap, not a mechanism gap (the cascade has no `wip/` scrub; squash-merge removes `wip/` by design), and the GitHub-issue home reuses the `gh issue create` pattern `/plan` and `/roadmap` already own without widening `/execute`'s closed write-target set — consistent with "author with skills, check with validate." D6 folds title + two-part body into the single existing `gh pr edit` in `pr_finalization` (no new state, no new write target), with the title derived from the validated slug (never raw prose) and `pr-creation/SKILL.md` as canonical spec. Both minimal and consistent with the stated drivers.

- **Cross-cutting contract preservation (PASS):** The design preserves `/execute`'s metadata-only inspection (D3 explicitly keeps the content read in `/plan`, never `/execute`) and the closed write-target set (D5 deliberately defers the automated emit precisely to avoid adding a remote write target; D6 reuses the existing `gh pr edit`). The single-pr-focus scoping is clearly bounded: Consequences and the PRD Out-of-Scope both state the coordinated path's finalization contract is unchanged and a later effort is deferred. No decision contradicts another at the architectural level: D1's adopt-PR and D2's DRAFT-before-READY compose (the adopted PR is readied by `gh pr ready` in `plan_completion`, after the pause boundary); D3's self-application (the feature setting `user_visible_surface: true`) is consistent.

- **CONCERN — D2 surface naming contradiction (prose defect, not architectural):** D2 explicitly REJECTS an explicit `--pause-for-review` flag (option (b), lines 131-134: "Rejected (developer directive): the implement-then-pause behavior must be tied to interactive mode") and chooses mode-driven resolution ("not a flag", lines 121-123). Yet the Decision Outcome line 229 names "the `--pause-for-review` flag" as the user-visible surface this feature dogfoods: "This feature itself adds user-visible surface (the `--pause-for-review` flag and the adopt-PR behavior)". The flag does not exist in the chosen design — this is a leftover from the decision-2 report's superseded recommendation. The dogfooded user-visible surface is actually the interactive-vs-`--auto` pause BEHAVIOR plus the adopt-PR behavior, not a flag. The contradiction is localized to one sentence and does not affect the mechanism, the R-mapping, or any state-machine wiring, so it does not sink the design — but it should be corrected before /plan decomposes, because /plan reads this body and could emit a docs work item for a flag that will never ship.

## Required changes

None blocking. Recommended (non-blocking) fix: line 229 — replace "the `--pause-for-review` flag" with the actual chosen surface, e.g. "the mode-driven pause behavior (interactive pauses, `--auto` finalizes)", to remove the contradiction with the D2(a)/(b) decision that rejects the flag. Without this fix the design is still implementable as written (the mechanism sections are unambiguous), but the dogfood claim points at a non-existent surface.

## Summary

The design satisfies every PRD requirement (R1–R8) with a coherent, minimal, source-grounded decision per requirement, preserves `/execute`'s metadata-only and closed-write-target contracts, and the mode-driven pause is a sound — arguably superior — choice over the decision report's rejected flag because it makes the R8-forbidden `--auto`+pause combination unrepresentable. The one real defect is a localized prose contradiction (line 229 names a `--pause-for-review` flag that D2 explicitly rejected); it is a one-sentence fix that does not compromise architectural soundness. PASS.
