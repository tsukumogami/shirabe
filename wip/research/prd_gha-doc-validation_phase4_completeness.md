# Completeness Review

## Verdict: FAIL
The PRD has strong structural coverage but leaves implementers without enough specificity on several requirements to build without guessing, and at least four requirements lack corresponding acceptance criteria.

## Issues Found

1. **R9 (custom-statuses) behavior is underspecified — extends vs. replaces**: The requirement says the input "extends or replaces" the canonical enum but doesn't define when each applies. An implementer must guess whether `custom-statuses: {prd: [Draft, Delivered]}` means the canonical PRD enum is replaced (so `Accepted` and `Done` now fail) or augmented (so `Delivered` is accepted in addition to all canonical values). The AC only tests that a custom value passes, not that canonical values remain valid or are excluded. Fix: add a sentence specifying the default behavior (replace, not extend) and add an AC verifying that a canonical-only value fails when the override is replace-mode.

2. **R7 (public-repo VISION check) has no mechanism for the "blocking vs. advisory" question raised in the scope**: The scope document explicitly flagged "Blocking vs. advisory" as a coverage note requiring an explicit requirement. The PRD says the check "fails CI" (implying blocking) but never makes this explicit at the requirement level, nor does it define whether this can be toggled. If a repo wants advisory-only runs (non-blocking), there's nothing here to address that, and no requirement for how the workflow job exits. Fix: add an explicit requirement or a sentence to R3/R11 clarifying that the static tier always exits non-zero on violations (blocking) and that advisory mode is out of scope for v1.

3. **R6 (Plan structural rules) lacks an AC for the `schema` field mismatch**: The AC checks `schema: plan/v2` (invalid), but R6 also requires that when `upstream` is present the referenced file must exist on disk and be tracked by git. The "tracked by git" clause has no AC. An existing-on-disk file that isn't tracked would silently pass. Fix: add an AC: "A Plan doc with `upstream:` pointing to a file that exists on disk but is not tracked by git fails CI."

4. **R11 (GHA annotations) — line number is required but format detection is silent on non-YAML frontmatter**: The annotation format specifies `line=<N>`, but what line number is emitted for FC04 (missing section headings)? Sections are absent by definition — there's no line to point to. An implementer will encounter this ambiguity and have to invent a convention. Fix: specify that for FC04 (missing section) the annotation uses line 1 (or the last line of the frontmatter block), or explicitly say "line is omitted for FC04."

5. **R14 (stable job name) has no AC**: The requirement states the job name must not change within a major version and that downstream repos hard-code it in branch protection. There is no acceptance criterion verifying the job is named consistently, nor a statement of what the canonical job name is. Fix: add an AC such as "The reusable workflow exposes a job named `validate-docs`; the workflow YAML at `@v1` uses this name." Also name the job explicitly in R14.

6. **R15 (fast execution) has no AC with a measurement methodology**: The AC for timing says "completes in under 60 seconds for a PR touching 5 doc files" but doesn't specify how this is measured (wall-clock in CI? local timing?). For a non-functional requirement, this is fine for a first draft, but for acceptance sign-off the method needs to be defined — especially since CI runner speed is variable. Fix: add a note that this is measured as workflow wall-clock time in a GitHub-hosted ubuntu-latest runner, and that it is validated by inspection of a CI run in shirabe's own repo.

7. **No requirement or AC covers the Roadmap format's reserved section rule mentioned in the scope**: The scope document's "In Scope" section lists "reserved section presence in Roadmap" as a static check. The PRD's R5 shows Roadmap required sections, and R7 covers VISION's reserved (excluded) sections, but there is no analog requirement for Roadmap-specific section exclusions. This is either a gap or needs an explicit statement that Roadmap has no excluded sections. Fix: either add a requirement for Roadmap reserved-section exclusions (consistent with the scope), or add a sentence to R5 stating Roadmap has no reserved (excluded) sections.

8. **Runner billing clarification in scope diverges from PRD's Known Limitations**: The scope document's "Coverage Notes" section says "all compute for downstream repos' validation runs against shirabe's GitHub Actions minutes" and asks whether v1 needs a solution. The PRD's Known Limitations correctly inverts this ("Compute runs in the caller's account") but doesn't explicitly state whether any documentation or requirement addresses this for private repos. The scope's concern was about cost management, and the PRD drops it without a clear disposition. Fix: add a sentence to Known Limitations stating that no v1 requirement addresses cost management for private downstream repos, as public repos use free runners and private repo costs are negligible given the 60-second execution bound.

9. **No AC covers the "no files match any prefix" case producing a notice (R10)**: The AC states "A PR that touches no doc files exits 0 with a notice." However, R10 specifies the trigger as "no files in the PR match any known format prefix," which is different from "touches no doc files" — a PR could modify `docs/README.md` (a doc file with no format prefix) and trigger the notice. The AC wording conflates the two. Fix: restate the AC to match R10's language: "A PR that modifies only files not matching any format prefix exits 0 and emits a workflow notice."

## Suggested Improvements

1. **Add a worked caller example to AC**: US1 mentions a single PR with one workflow file. The AC shows the caller YAML snippet, but it uses a `paths` filter that limits validation to `docs/**`. If a downstream repo stores docs at a different path, the caller template silently skips them. Adding a note or an AC variant without the path filter — or explicitly stating the path filter is the caller's responsibility and not part of the reusable workflow — would prevent a common misuse.

2. **Define a "format not recognized but name looks structural" behavior**: R8 says files not matching any prefix are skipped silently. A file named `PLAN_foo.md` (underscore instead of hyphen) would be skipped. Consider adding a note that this is intentional or adding a warning annotation for filenames that are close matches (not required for v1, but worth calling out as a future improvement).

3. **Clarify when R16 (branch protection) takes effect**: The AC says "shirabe's branch protection includes `validate-docs` as a required status check," but the workflow must be merged before it can be set as a required check. This creates a bootstrapping chicken-and-egg that the PRD doesn't acknowledge. Noting the deployment sequence (merge first, then update branch protection) would help the implementer.

4. **Version tag strategy deserves a sentence in R1 or AC**: The AC says "tagged `v1` on initial release" and that `@v1` receives patch improvements automatically. This implies a floating major-version tag strategy (e.g., the `v1` tag is a moving pointer, not a static ref). This is standard practice but should be explicitly stated in R1 or a Decisions section entry so it doesn't get implemented as an immutable tag.

5. **FC03 check needs a case-sensitivity note**: R4 defines canonical status values with specific casing (e.g., "In Progress", "Accepted"). FC03 compares frontmatter `status` with the `## Status` section body. If the body reads `**Status:** accepted` (lowercase), does FC03 fire? Define whether comparisons are case-sensitive or case-normalized.

## Summary

The PRD is well-structured and covers the problem space clearly. The Decisions and Trade-offs section is particularly strong, and the Out of Scope list is specific enough to prevent most scope creep. However, nine issues — ranging from missing ACs for existing requirements (R6 upstream-tracked-by-git, R14 job name, R10 notice wording) to underspecified behavior (R9 extend vs. replace, R11 line numbers for absent sections) to a gap carried from the scope document (Roadmap reserved sections) — mean an implementer would need to make judgment calls that should be design decisions. The PRD should reach PASS after resolving issues 1, 2, 5, and 7 at minimum; the remaining issues are lower risk but worth addressing before handoff.
