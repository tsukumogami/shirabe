# Clarity Review

## Verdict: PASS

The PRD is unusually precise — most requirements name exact file paths, exact filenames with templated slots, exact state-file fields, and binary AC checks. The few ambiguities are bounded and either explicitly deferred (Open Questions) or load-bearing on an external decision; they do not let two implementers ship divergent builds.

## Ambiguities Found

1. **R4 / AC6 — "thesis-shift signal" detection.** Lines 284-286, 574-576: "The author's Phase 1 discovery surfaces a thesis-shift signal (author-stated; the brief calls this out as 'if the long-term thesis is shifting')." -> The detection mechanism is "elicits the signal conversationally" with no enumerated trigger keywords, prompts, or example author phrasings. Two implementers could ship different Phase 1 prompts and disagree on what counts as a thesis-shift signal. -> Either (a) enumerate 2-4 example author phrasings the Phase 1 prompt must surface to, or (b) specify the exact Phase 1 prompt text that elicits the signal, or (c) explicitly defer signal-detection wording to design with a named acceptance gate (parallels Open Question 6 for competitive-framing signals).

2. **R7 — "Coordination Dependencies section names cross-block dependencies."** Lines 324-328: The gate for invoking `/roadmap` is "Building Blocks count is 3+ AND Coordination Dependencies section names cross-block dependencies." -> "Names cross-block dependencies" is ambiguous: does an empty bulleted section count? a section with one dependency? a section that exists but says "none identified"? AC9 says "no Coordination Dependencies section" skips, but R7 says "no explicit Coordination Dependencies." -> Define precisely: e.g., "Coordination Dependencies section contains ≥1 non-empty entry referencing another Building Block by name." Update AC9 and AC10 to match.

3. **R11 / AC20 — child suppression signaling.** Lines 437-443, 647-651: "`/charter` MUST decide upfront whether the re-entry path is a re-evaluation exit ... or a fresh chain (signal the child to suppress its status-aware re-entry). The signaling mechanism is a design-team decision; the requirement is that `/charter`'s flow MUST NOT be hijacked by a child's resume-time prompt." -> AC20 says "mechanism is design-team scope; AC verifies observable outcome" but does not state the observable outcome that gets verified. Two reviewers running AC20 could disagree on whether the test passed. -> State the observable outcome explicitly: e.g., "AC20 passes if no child-emitted 'offer to revise or start fresh' prompt reaches the author during a `/charter`-driven re-entry; verified by transcript inspection of the chain run."

4. **R15 — "ADR-style body shape."** Lines 497-515: Decision Records "MUST follow the ADR-style body shape (Status, Context, Decision, Options Considered, Consequences) with frontmatter (`status`, `decision`, `rationale`)." -> The body sections are named but their content requirements are only partially specified (per-sub-shape Context/Decision/Options-Considered are constrained; Consequences is not). Frontmatter field value formats are unspecified (`status` enum? `decision` free text? `rationale` length bound?). -> Either (a) specify the Consequences section's content rule and frontmatter value formats, or (b) point to a fixture/example Decision Record that ships with the skill as the canonical shape.

5. **R10 — `chain_skipped` reasons format.** Lines 399-400: "`chain_skipped: [<sub-list of skipped children with reasons>]`" -> The reason format is unspecified. Free text? Enumerated codes? Does the resume ladder parse these reasons? -> Specify either an enum of allowed reasons (e.g., `comp_not_shipped`, `no_thesis_shift`, `insufficient_building_blocks`) or declare reasons as free-text-for-humans only.

6. **R10 — state file schema "YAML front-matter style".** Lines 392-416: The state file is described as "YAML front-matter style" but no opening/closing `---` delimiters are shown in the example block, and the file is `.md` (not `.yaml`). Is the file pure YAML, or markdown with YAML frontmatter and a body? -> Specify exactly: either "the entire file is YAML" or "markdown file with YAML frontmatter; body is free-form notes."

7. **R11 — last_updated comparison for staleness.** Lines 425-431, 642-645: AC19 compares `child_snapshots.strategy.last_seen` against "the STRATEGY's current frontmatter timestamp." -> The STRATEGY frontmatter field name carrying the timestamp is not specified (is it `last_updated`? `updated`? `accepted_at`?). If the STRATEGY format does not currently expose a comparable timestamp, the comparison is unimplementable. -> Name the STRATEGY frontmatter field, or specify the file mtime as the comparison source, or flag this as a STRATEGY-format dependency in Open Questions.

8. **R5 / AC8 — "chain proposal output is identical to a public-repo invocation."** Lines 581-583: AC8 verifies that in a private repo without `/comp`, "the chain proposal output is identical to a public-repo invocation." -> "Identical" is precise, but the *chain proposal output* is not defined anywhere in the PRD as a distinct artifact. Is it a phase-1 console message? A state-file field? A specific prompt? -> Define "chain proposal output" once, then AC8's identity check becomes binary.

9. **R16 / AC17 — "stale-session detection fired on resume" vs. "≥ 7 days."** Lines 192-194, 633-636: US-3b describes the bail trigger as either (a) "wrap it up" intent or (b) "state file's `last_updated` is > 7 days old." AC17 uses "≥ 7 days." R16 says "7-day stale-session threshold." -> The boundary at exactly 7 days is inconsistent (`>` vs `≥`). Pick one.

10. **R8 — "most-recently-running child's intermediate."** Lines 372-376: Abandonment-forced "force-materialize[s] the most-recently-running child's intermediate as a Draft artifact." -> "Most-recently-running" is unambiguous when one child was active; ambiguous when the state file records two partial children (e.g., `/vision` Phase 3 partial intermediate exists AND `/strategy` Phase 1 started). The `planned_chain` and `chain_ran` fields imply a sequence, but no tie-break rule is stated. -> Specify the tie-break: "most-recently-running = the last entry in `chain_ran`, or if `chain_ran` is empty, the first entry in `planned_chain` with a non-empty wip/ intermediate."

## Suggested Improvements

1. **Add a glossary of load-bearing terms.** Rationale: "thesis-shift signal," "chain proposal," "intermediate," "review surface," "durable artifact," "discipline-vs-artifact decoupling" are all used as precise terms but defined inline at first mention only. A 6-8-entry glossary at the top of the PRD (or in Open Questions if some definitions are deferred) anchors the AC vocabulary and prevents drift between reviewer interpretations.

2. **Promote AC observable-outcome rules to first-class.** Rationale: AC20 explicitly says "AC verifies observable outcome" but does not state which observable to inspect. Apply the same lens to AC6, AC7, AC8: each should name (a) the verification command or inspection target and (b) the exact pass condition. This makes the jury-clarity bar testable rather than aspirational.

3. **Cross-link the R7 STRATEGY-path open question into the relevant AC.** Rationale: Open Question 1 flags that R7's delegation contract may need renegotiation if `/roadmap` rejects STRATEGY paths. AC10 verifies the happy path but does not name a fallback AC for the rejection case. Add either "AC10 is contingent on Open Question 1 resolving in favor of acceptance" or a sibling AC10b for the fallback path.

4. **State the AC verification harness once.** Rationale: ACs reference `shirabe validate`, `scripts/run-evals.sh`, and "transcript inspection" implicitly. A single line near the AC heading naming the verification entry points ("All ACs are verified by [eval scenarios | shirabe validate | manual review of generated state file]") removes one round-trip during Phase 4 reviewer cycles.

## Summary

The PRD is well-scoped and unusually concrete for its size — file paths, filenames, state-file schemas, and chain-shape gates are nearly all binary-verifiable. The clarity gaps that remain cluster in three areas: detection-of-signals (thesis-shift, competitive-framing) deferred to agent judgment without enumerated prompts; AC observable-outcome rules that say "verify the outcome" without naming the outcome; and a handful of off-by-one or format ambiguities (`>` vs `≥` on the stale threshold, `chain_skipped` reasons format, state file YAML shape). None of these are load-bearing enough to fail the PRD — they are revision targets the author can address in a final-pass tightening before Accepted status.
