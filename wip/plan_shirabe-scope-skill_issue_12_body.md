## Goal

Ship four Decision Record body templates at `skills/scope/references/decision-record-{prd|design}-{re-evaluation|rejection}.md` so `/scope` Phase 3 finalization can populate them deterministically when the chain exits via the re-evaluation or rejection sub-shape at either boundary. Each template (~50-80 lines) mirrors `/charter`'s ADR-style body shape (Status / Context / Decision / Options Considered / Consequences) with `/scope`-specific framing for the two boundary positions (PRD-Accepted held, DESIGN-Accepted held) and the two sub-shapes (re-evaluation, rejection).

## Context

Design: `docs/designs/DESIGN-shirabe-scope-skill.md`

Interface I.2 (lines 1710-1723) defines the Decision Record path schema: `docs/decisions/DECISION-{prd|design}-<topic>-{re-evaluation|rejection}-<YYYY-MM-DD>.md`. Two boundary positions × two sub-shapes = four combinations. All four share the ADR-style body shape per PRD R15 with frontmatter (`status: {Draft | Accepted}`, `decision:`, `rationale:`); per-combination body templates live at the canonical path above so `/scope` Phase 3 can copy + substitute at runtime without re-deriving each body's structure.

Decisions 1 and 5 establish what each template references and what it must NOT reference:

- **Decision 1 (lines 286-391)** settles the Phase-N Reject contract placement on `/prd` Phase 4 and `/design` Phase 6. The Reject branch produces a discard commit (`docs(prd): discard PRD draft for <topic>` or `docs(design): discard DESIGN draft for <topic>`) and removes the durable artifact via `git rm`. On in-chain Reject, `/scope` writes the rejection-sub-shape Decision Record referencing the discard commit SHA (durable trace), NOT the absent artifact (which no longer exists on disk).
- **Decision 5 (lines 731-794)** settles cross-boundary state-snapshot semantics: when `/scope` writes a re-evaluation Decision Record, `child_snapshots:` stays frozen on the referenced upstream artifact (the existing PRD or DESIGN). The re-evaluation template's Consequences section MUST reference the existing PRD/DESIGN at its canonical path (substituted from the state file's `referenced_artifact:` field) so drift detection's job — comparing the snapshot to the current artifact — stays well-defined on subsequent `/scope` resumes.

Component 7.7 (lines 1627-1652) anchors the rejection-sub-shape template's load-bearing claim: `/scope` observes the Reject verdict by `git log`-searching for the discard commit between the `parent_orchestration:` write and the child return. The rejection template's Context section MUST reference the discard commit SHA inline so the Decision Record carries the durable trace forward (per AC30c parity between in-chain and out-of-chain Reject).

This issue mirrors `/charter`'s precedent. `/charter` ships two Decision Record templates at `skills/charter/references/templates/decision-record-{re-evaluation|rejection}.md` for its single STRATEGY boundary. `/scope` has two boundaries (PRD and DESIGN), so four templates ship — twice the count, same structural shape. Each `/scope` template carries the same five-section body (Status / Context / Decision / Options Considered / Consequences) with frontmatter on top; the per-template prose differs in which boundary's child the template attaches to and which sub-shape (held vs rejected) the template describes.

The four template files are parallel single-file artifacts with no cross-template behavioral coupling. Each template is consumed by `/scope` Phase 3 (Issue 11's `phase-3-exit-finalization.md`) which selects the appropriate template at runtime based on `exit:` + `boundary:` + `decision_record_sub_shape:` from the state file. The templates' canonical paths are what Issue 11's phase-3 prose will cite by name.
