# Private Repository Content Guidelines

You are creating content for a **private repository**. Apply these guidelines.

## General Principles

- Write for team members who understand the project context
- Internal shorthand and references are acceptable
- Focus on decision rationale and trade-offs
- Document "why" decisions were made, not just "what"

## What's Allowed

- Reference competitors by name in analysis and comparisons
- Include internal business rationale and stakeholder discussions
- Reference exploratory ideas and approaches that may not pan out
- Assume reader knows organizational context
- Reference issues, PRs, and designs in any repo (private or public)
- Link to private repos and internal resources
- Include pre-announcement feature discussions
- Document abandoned approaches for historical context

## By Artifact Type

### Design Documents
- Include competitor analysis with names and direct comparisons
- Document internal stakeholder discussions and decisions
- Include exploratory thinking and ideas that may not pan out
- Reference internal business rationale for decisions
- Can reference designs/issues in any repo

### Issues
- Can mention competitors by name in analysis
- Can include internal business rationale
- Can reference exploratory ideas that may not pan out
- Can assume reader knows internal context
- Can reference issues in any repo (private or public)

### Pull Requests
- PR body can include internal rationale and context
- Can reference competitor analysis or internal discussions
- Can mention exploratory approaches considered
- Commit messages can include internal context
- Can reference issues/PRs in any repo

### Code Comments
- Code comments can assume team knowledge
- Can reference internal decisions or rationale
- TODOs can mention internal tracking systems
