# Phase 5: Finalization

Create implementation summary, clean up artifacts, and perform final verification.

## Resume Check

If `wip/issue_<N>_summary.md` exists and cleanup commit exists, skip to Phase 6.

## Auto-Skip Logic

**Summary generation auto-skips for simple issues.**

Check your project's label vocabulary (defined in `## Label Vocabulary` in CLAUDE.md)
for which labels indicate summary-worthy vs summary-skippable work.

**Default behavior** (when no label vocabulary is defined):
- Skip summary for issues labeled `docs`, `documentation`, `config`, or `chore` (without `bug`/`enhancement`)
- Generate summary for issues labeled `bug`, `enhancement`, `refactor`, or `security`

**User override:** User can request "skip the summary" or "I want a summary" regardless of labels.

**When skipping:** Execute steps 5.1 (Code Cleanup), 5.2 (Final Verification), 5.6 (Final Test Run), and 5.7 (Consider Manual Testing). Skip steps 5.3 (Create Summary), 5.4 (Commit Summary), 5.5 (Cleanup Artifacts with separate commit). The requirements mapping table is also skipped for these issue types. Instead, clean up `wip/` and commit once after final verification.

**When generating:** Execute all steps 5.1-5.7 as normal.

## Steps

### 5.1 Code Cleanup

Remove development artifacts:
- Debug statements (language-specific print/log calls used for debugging, not production logging)
- Commented-out code
- TODO comments that were addressed
- Unused imports

### 5.2 Final Verification

Run complete test suite:
- All unit tests
- Integration tests (if applicable)
- Build verification
- Linting

All checks must pass.

### 5.3 Create Summary

Create `wip/issue_<N>_summary.md` (Write tool):

```markdown
# Issue <N> Summary

## What Was Implemented
<Brief description of the solution>

## Changes Made
- `path/to/file1`: <what changed>
- `path/to/file2`: <what changed>

## Key Decisions
- <Decision 1>: <rationale>
- <Decision 2>: <rationale>

## Trade-offs Accepted
- <Trade-off>: <why acceptable>

## Test Coverage
- New tests added: <count>
- Coverage change: <before> → <after>

## Known Limitations
- <Limitation 1>
- <Limitation 2>

## Future Improvements
<Optional: things that could be improved later>

## Requirements Mapping

| AC | Status | Evidence / Reason |
|----|--------|-------------------|
| <criterion from issue> | Implemented | <file:function or line range> |
| <criterion from issue> | Deviated | <what was done instead and why> |
```

### 5.4 Commit Summary

Commit using format: `docs: add implementation summary`

This preserves the summary in git history for future reference.

### 5.5 Cleanup Artifacts

Delete temporary files:

```bash
# Remove wip/ directory (includes context file from Phase 0)
rm -rf wip/

# Remove any coverage output files generated by your project's test commands
# (e.g., coverage.out, coverage.html, .coverage, lcov.info — varies by language)
```

Commit using format: `chore: clean up temporary artifacts`

Note: The baseline, plan, and summary are preserved in git history via their commits. Deleting the files keeps the working tree clean while maintaining full audit trail.

### 5.6 Final Test Run

Run complete test suite one more time to verify cleanup didn't break anything.

### 5.7 Consider Manual Testing

Evaluate whether manual testing would add value before creating the PR. Consider recommending `/try-it` if:

**Recommend when:**
- Changes affect user-facing behavior or CLI commands
- Complex logic or edge cases were added
- Integration between components might have subtle issues
- Error handling or validation was modified
- Changes could have unexpected side effects
- Automated tests might not catch certain classes of bugs

**Skip when:**
- Documentation-only changes
- Configuration files with no runtime impact
- Changes already thoroughly covered by existing tests
- Simple refactoring with no behavior changes
- Internal implementation details that don't affect outputs

If you recommend `/try-it`, inform the user: "Changes complete. Consider running `/try-it` to manually test the changes before creating the PR."

Let the user decide whether to invoke it.

## Success Criteria

- [ ] No debug code remaining
- [ ] All tests pass
- [ ] Build succeeds
- [ ] Summary committed
- [ ] Artifacts cleaned up
- [ ] Final verification passed
- [ ] Manual testing considered (if applicable)
